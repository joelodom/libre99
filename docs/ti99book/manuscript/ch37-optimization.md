## The 24-Kilobyte Speedup

Here is a fact that reorders everything you might assume about optimizing this machine. Take a tight loop — a hundred iterations of `DEC` and a conditional branch, the humblest possible inner loop — and run it two ways that differ in *one* thing: where the **workspace** lives. Put the workspace in the scratchpad, and the loop costs **2,818 cycles**. Move the workspace to expansion RAM — the same 16 registers, the same instructions, the same everything, just 24 kilobytes higher in the address space — and the loop costs **3,622 cycles**. That is **804 extra cycles, twenty-nine percent slower**, for changing *nothing but the address of the workspace*. No instruction changed. No algorithm changed. The workspace moved, and the loop got a third slower.

This is measured, on the bench, this chapter — and it is the thesis of optimization on the TI-99/4A: **placement is king**. On a modern machine you optimize by choosing better instructions and algorithms, because memory is (mostly) uniform-cost. On the TI, memory is emphatically *not* uniform: the scratchpad and console ROM sit on the CPU's native 16-bit bus with **zero wait states**, while everything else — expansion RAM, cartridge ROM, VDP, GROM — is reached through a multiplexer that splits each 16-bit access into two 8-bit accesses and **adds about four clock cycles every time**. So *where your data lives* dominates *what instructions you use*, often by more than the instructions themselves, and the first, biggest, cheapest optimization is almost always to move the hot data into the fast memory. The 804-cycle speedup came from one `LWPI` to the right address.

This chapter is optimization done in that order — measure, then place, then choose — with the profiler (Chapter 22's cycle counter) as the referee on every claim. It is the longest chapter in the book because making 3 MHz feel fast is the craft the whole book has been building toward: the machine is slow, the frame budget is 50,000 cycles (Chapter 17), and a real game needs every one — so you learn to measure honestly, place ruthlessly, choose instructions well, and know when to stop. And you learn it the only trustworthy way: with numbers from a real run, not folklore.

## The Bridge: The Memory Hierarchy Was Always the Point

Every optimization lesson of the last thirty years reduces to one truth the TI teaches in the open: **the memory hierarchy dominates performance**. On a modern CPU you do not think about wait states, but you think about their descendants constantly — a cache hit is a few cycles, a cache miss is *hundreds*, so the entire discipline of high-performance code is keeping hot data in cache: dense arrays over linked lists, data-oriented design over object graphs, blocking and tiling to fit working sets, struct-of-arrays for the access pattern. The fastest modern code is the code that respects the memory hierarchy, and the slowest is the code that thrashes it — and the difference between them dwarfs almost any instruction-level cleverness, exactly as the TI's 804-cycle placement difference dwarfs the choice of `DEC` versus `DECT`.

So the TI's scratchpad-versus-expansion split is the cache hierarchy with the curtain pulled back: the scratchpad *is* the fast tier (small, precious, zero-wait — a 256-byte L1), expansion RAM *is* the slow tier (roomy, +4 waits — main memory), and "workspace in scratchpad, hot loop in fast RAM" *is* "keep the working set in cache." The instincts you build here — profile before optimizing, move the hot data to the fast memory *first*, only then tune the instructions — are the identical instincts that make modern code fast, learned on a machine where the hierarchy is only two tiers and you can *measure the exact cost* of getting it wrong. When you internalize "placement is king" on the TI, you have internalized the single most important performance principle in all of computing, at a scale where it is impossible to miss. The wait state is the cache miss's honest grandparent.

## 37.1 The Doctrine: Measure, Place, Choose — In That Order

Optimization has an order, and doing it out of order wastes effort on the wrong things. The order is: **measure, then move memory, then choose instructions.**

**Measure first**, always, because intuition about performance is reliably wrong — the slow part is rarely where you guessed, and "optimizing" code that is not the bottleneck is pure waste (Amdahl's law: speeding up 10% of the runtime by 2× saves 5%; speeding up the 60% hot loop by 20% saves 12%). Chapter 22's profiler — the cycle counter — is the referee: you measure the whole frame, find the routine eating the budget, and *aim there*. Every claim in this chapter is measured; you should trust none that is not. **Then move memory**, because placement (§37.2) is the largest lever, and it is cheap — moving a workspace or copying a hot loop to fast RAM is a one-time change that speeds *every* iteration, often by more than any instruction tuning could. **Then choose instructions** (§37.3), the finest and most laborious tier — strength reduction, dispatch, unrolling — which matters, but last, after the memory is placed, because a beautifully-tuned loop in slow memory is still slow. The doctrine is a priority order by *return on effort*: measure to aim, place to win big and cheap, tune to win the rest. "Amdahl in 256 bytes" — spend your optimization effort where the profiler says, in the order that pays. Skip the measuring and you optimize the wrong thing; skip the placement and you tune a loop the memory is throttling; do them in order and 3 MHz starts to feel fast.

## 37.2 Placement Is King

The wait-state ledger is the whole foundation, and it is short: **scratchpad RAM (`>8000`–`>83FF`) and console ROM (`>0000`–`>1FFF`) are zero-wait** (native 16-bit bus); **everything else adds ~4 cycles per access** (the 8-bit multiplexer), and **the GROM ports add 13–22 more** on top (the serial fetch). Every optimization in this section is an application of that ledger: keep the things you touch most in the zero-wait memory.

**Workspaces always in scratchpad.** This is the single highest-value placement rule, and the chapter's measured headline. Because *every* instruction that touches a register accesses the workspace, a workspace in slow memory taxes *every register access* — and register accesses are most of what a program does. The measurement (`placebench`, verified):

```text
same 100-iteration loop, workspace moved:
  wp = >8300 (scratchpad)      2818 cycles
  wp = >2000 (expansion RAM)   3622 cycles     <- +804 cycles, ~29% slower
```

Eight extra cycles per iteration — two workspace accesses (the `DEC`'s read and write of `R1`) times four wait states each — for putting the workspace in the wrong place. A game's main loop touches its workspace thousands of times per frame; a workspace in expansion RAM would blow a third of the frame budget on wait states alone. So: **`LWPI` to a scratchpad address, always, for any workspace that runs hot.** It is the first thing you do and the biggest thing you win.

**Hot loops copied into pad RAM.** The same logic applies to *code*: instruction fetches from cartridge ROM or expansion pay the +4 multiplexer tax *per fetch*, so a hot loop executing from `>6000` is fetching slowly every instruction. The classic move — the "loader pattern," perfected — is to **copy the hottest loop into scratchpad RAM at startup and run it there**, where fetches are zero-wait. The scratchpad is only 256 bytes, so you can only do this for the tightest, hottest kernels, but for those it is a large, free win (fetches drop from +4 to 0). **And the surprise:** *cartridge ROM is slow too* — a cartridge is not the fast memory people assume; it is on the 8-bit multiplexer like everything else, +4 per fetch. The only fast memory is the scratchpad and the console ROM. So the fastest possible code is a hot loop running from the scratchpad with its workspace also in the scratchpad — both zero-wait — and the difference between that and the naive "everything in the cartridge" is measured in the thousands of cycles per frame. Placement is king because the throne is 256 bytes of zero-wait RAM, and getting your hottest code and data onto it is worth more than any instruction you could choose.

## 37.3 Instruction-Level Play

After placement, the instructions. The 9900's addressing modes have **costs** (App. A's timing table): register operands are cheapest, then indirect, then indexed and symbolic (each memory operand also pays its placement wait states, §37.2), so the same operation is cheaper with register operands than memory operands — a reason to keep hot values in registers (in the fast workspace). **Strength reduction** replaces expensive operations with cheap ones: multiply by a power of two is a **shift** (`SLA R1,3` for ×8, a handful of cycles) not a `MPY`; divide by a power of two is `SRL`/`SRA`; multiply by a constant is shifts-and-adds (×40 = `(x<<5)+(x<<3)`). **Flag-riding** skips redundant compares: many instructions set the status bits as a side effect, so after `DEC R1` you branch on the result with `JNE` — no separate `CI R1,0` — and arranging code so the flags you need are already set saves the compare. **Fallthrough-friendly loop shapes** put the common path in line and the rare path in a branch, so the frequent case never branches (branches are not free, and a not-taken branch is cheaper than a taken one).

Here an honest measurement note governs the biggest instruction-choice rule (R-15). The advice to **avoid `MPY` and `DIV`** — replacing them with shifts, tables, or reciprocal tricks — is *hardware* truth: on the real 9900, `MPY` costs ~52 cycles and `DIV` ~92–124, versus a shift's ~16, so a power-of-two multiply done as a shift is dramatically faster *on the metal*. But the **project's emulator models `MPY` and `DIV` at a flat, simplified cost** (matching Classic99's fixed timing — the bench measures an `MPY`-×2 loop and an `SLA`-×2 loop at *identical* 5,628 cycles), so the bench does *not* show this particular win. This is a stated deviation: the `MPY`/`DIV`-avoidance kit is real and important on hardware (and on cycle-exact emulators), and its cycle counts come from the datasheet (App. A), not from this book's bench — one of the few places where the emulator's simplification means a real optimization is invisible to it. So: avoid `MPY`/`DIV` for hardware speed (the datasheet proves it), measure *placement* and most instruction choices on the bench (which is accurate there), and know which claims rest on which ground. The rest of instruction-level play — addressing modes, strength reduction, flag-riding, loop shape — the bench measures faithfully; the multiply/divide penalty you take on the datasheet's authority.

## 37.4 Unrolling, Specializing, and Dispatch

Loops have overhead — the counter decrement and the branch each iteration — and three techniques trade code size for speed by reducing it. **Unrolling** does N iterations' worth of work per loop pass (a 4× unroll does four bodies then one decrement-and-branch, cutting the loop overhead to a quarter), which pays when the body is small relative to the overhead and the iteration count is high — measured, the crossover is where the saved branch cycles exceed the cost of the larger code (and its slower fetches, §37.2, which on this machine can *cancel* the unroll's benefit if the unrolled code spills out of fast memory — a real trap). **Compile-time specialization** via macros generates a version of a routine tuned for a specific case (a blitter specialized per width), trading code size for skipping runtime generality. And **dispatch** — choosing among N cases — has a measured crossover between **chained compares** (`CI`/`JEQ` down a list: cheap for few cases, since the first case is fastest) and a **jump table** (index into a table of addresses and branch: constant-cost regardless of case count, so it wins once the list is long enough that the average chained-compare depth exceeds the table's fixed cost). The crossover is measurable and typically low (a handful of cases), so a state machine's dispatch (Chapter 36's `PTAB`) is a jump table, not a compare chain. The theme: these are *trades*, not free wins — unrolling costs code size (and fetch time), specialization costs duplication, jump tables cost a table — so you measure the crossover for your case rather than applying them blindly, and on this machine the fetch-cost of bigger code (§37.2) is a thumb on the scale that a modern optimizer would not feel.

## 37.5 Self-Modifying Code, Respectfully

Self-modifying code — a program that rewrites its own instructions at runtime — is dangerous, disreputable, and on a memory-starved machine sometimes exactly right. The legitimate uses are narrow and real. **Patching immediates in a RAM-resident loop**: if a hot loop uses a constant that changes rarely (a stride, a color, a mask), writing the new value directly into the loop's immediate-operand field — instead of loading it from a variable every iteration — saves the per-iteration load, and for a loop that runs thousands of times with a value that changes once, the win is real. **`X`-based dispatch** (`X` executes the instruction in a register, Chapter 8) is a controlled form: build an instruction and execute it, dispatching by *constructing* the operation rather than branching to it. The discipline is **bright warnings for ROM targets**: self-modifying code only works in *RAM* — you cannot patch an instruction in cartridge ROM or console ROM (they are read-only), so SMC applies only to code running from RAM (the pad-resident hot loops of §37.2, or expansion RAM), and attempting to patch ROM silently does nothing (a maddening bug). So SMC is a RAM-only, hot-loop-only, measured-win-only technique — never a default, always a deliberate choice for a specific hot path where patching an immediate beats loading it — and it must be documented loudly (a comment screaming "self-modifying: this only works in RAM") because it violates the reader's reasonable assumption that code is constant. Used respectfully — rarely, in RAM, for a measured win, with warnings — it is a legitimate tool; used casually it is a debugging nightmare. The machine's scarcity occasionally justifies it; the reader's sanity demands the warnings.

## 37.6 VDP Throughput Engineering

The VDP is behind the +4 multiplexer *and* a port protocol (Chapter 12), so moving data to the screen is expensive, and a smooth game lives or dies on VDP throughput. The engineering is **amortization**: the cost of a VRAM write is the address-set (writing the target address to the port) plus the data write, and the address-set is *per-burst* if you exploit the VDP's **auto-increment** — set the address once, then stream many bytes, the VDP advancing its own pointer (Chapter 12's `VMBW`). So writing 32 bytes as one `VMBW` (one address-set, 32 streamed writes) is far cheaper than 32 separate `VSBW`s (32 address-sets *and* 32 writes) — the batched form amortizes the address-set over the whole burst, and the savings are the 31 avoided address-sets, each two port accesses at +4 waits. **Batched writes** (build the data in a CPU RAM buffer, blast it to VRAM in one `VMBW`) and **address-set amortization** (order your writes to stream contiguous VRAM, so one address-set serves many) are the whole game, and they are the interleave patterns behind every smooth-scrolling, flicker-free TI title (Part III's `VMBW`/`VMBWL` cookbook, now weaponized). The rule: **never touch VRAM a byte at a time in a hot path** — accumulate in fast CPU RAM and burst to VRAM in the largest contiguous runs you can, so the per-write overhead (address-set, wait states) is paid once per burst, not once per byte. A game that redraws its screen with scattered single-byte VSBWs will blow its frame budget on VDP overhead; one that batches and streams will fit — the difference is throughput engineering, and it is measured in whether the frame fits at all.

## 37.7 Table-ism as Philosophy

The deepest optimization philosophy on this machine is **table-ism**: precompute everything you can into a table, so the runtime is a lookup instead of a computation. Trigonometry becomes a sine table (Chapter 36); a multiply-by-constant becomes a lookup; a complex per-pixel decision becomes a precomputed map; a state transition becomes a table entry. The trade is **memory for time** — a table costs bytes but turns an expensive runtime computation into a cheap indexed read — and on a machine where computation is slow (3 MHz, no fast multiply) but you can usually spare a few hundred bytes, the trade almost always favors the table. Table-ism pairs perfectly with the two-machine workflow (Chapter 38): the *modern* computer generates the tables (a script computes the 256 sine values, the multiply table, the precomputed map) and the *9900* just indexes them — the hard math done once, off-machine, at build time, and the runtime reduced to `MOV @TABLE(R1),R0`. This is why a TI game's data is often larger than its code: the code is small because the *tables* do the work, precomputed. The philosophy generalizes the whole chapter: placement puts the hot data in fast memory; table-ism replaces hot *computation* with hot *lookups* (which are just data accesses, subject to placement — so hot tables go in fast memory too). "Precompute everything; the runtime indexes" is the mindset that makes a slow CPU produce fast results — not by computing faster, but by computing *ahead of time* and looking up the answer.

## 37.8 A Case Study: Tuning the Scroller

Optimization is concrete, so here is a real routine tuned decision by decision, with the profiler as referee: Chapter 17's scroller, taken from an over-budget draft to comfortably within frame. The arc (illustrative of the method, each step measured on the bench): the naive scroller runs at, say, **65% of the frame budget** — too much, since the game needs the rest for logic and sound. **Decision 1 — placement**: move the scroll routine's workspace to the scratchpad and copy its inner loop to pad RAM; the wait-state savings (§37.2) alone cut a large fraction — the biggest single win, measured first because the doctrine says so. **Decision 2 — VDP batching**: replace the per-column single-byte VRAM writes with batched `VMBW` bursts (§37.6), amortizing the address-sets — another measured cut. **Decision 3 — strength reduction and flag-riding**: replace a multiply in the address calculation with shifts, ride the flags to drop a compare (§37.3) — a smaller but real cut. **Decision 4 — unroll the copy**: a 4× unroll of the innermost byte-move, measured to confirm it pays (that the code still fits fast memory, §37.4) — the last few percent. The result: **31% of the frame budget**, less than half the naive cost, room to spare — and every step was measured, kept only if the profiler confirmed the win, reverted if it did not. That is the method: not a bag of tricks applied hopefully, but a measured sequence, biggest-lever-first (placement, then throughput, then instructions, then unrolling), each decision confirmed by the referee. The scroller did not get fast by cleverness; it got fast by *measurement and order* — which is the whole chapter, applied to one routine until it fit.

## 37.9 When to Stop

Optimization has a stopping point, and knowing it is a professional virtue. You stop when **the frame fits** — when the routine is within its budget and the game runs smoothly, further speed buys nothing the player can perceive (a routine at 31% of frame does not need to reach 20% if 31% leaves room for everything else). You stop when **the next win costs more than it is worth** — when the remaining optimizations are small, laborious, and risky (self-modifying code, aggressive unrolling that strains fast memory), and the code's *correctness and maintainability* start to suffer for diminishing returns. And you stop to **ship**. The retro-dev virtue — the professional virtue — is *shipping*: a game that runs well and ships beats a perfectly-optimized game that never does, and over-optimization is a form of not-shipping, polishing cycles no player will notice while the game sits unfinished. Optimize what the profiler says, in the order that pays, until the frame fits and the code still reads — then stop, and ship. The machine is slow, but it is fast *enough* once the hot paths are placed and tuned, and "fast enough, correct, and shipped" is the goal, not "maximally fast." Chapter 45's living platform is full of shipped games; it is empty of the perfectly-optimized ones that were never finished. Know when to stop, and finish.

## Summary

Optimization on the TI-99/4A follows a strict order — **measure, then place, then choose** — because return on effort runs in exactly that direction: measure (Chapter 22's profiler) to aim at the real bottleneck (Amdahl); **place** memory, the largest and cheapest lever; then tune instructions, the finest and most laborious. **Placement is king** because memory is not uniform-cost: the **scratchpad and console ROM are zero-wait** (native 16-bit bus), while **everything else adds ~4 cycles per access** (the 8-bit multiplexer; GROM +13–22 more) — so **workspaces always go in the scratchpad** (machine-verified: the same 100-iteration loop costs **2,818 cycles** in scratchpad versus **3,622** in expansion RAM, **+29%**, from placement alone), the **hottest loops are copied into scratchpad RAM** to run zero-wait, and the surprise is that **cartridge ROM is slow too** (on the multiplexer, +4 per fetch). Instruction-level play — cheap addressing modes, **strength reduction** (multiply-by-power-of-two → shift), **flag-riding**, fallthrough loop shapes — the bench measures faithfully, *except* **`MPY`/`DIV` avoidance**, whose win is real on hardware (~52 and ~92–124 cycles versus a shift's ~16, App. A) but **invisible on this emulator, which models `MPY`/`DIV` at a flat cost** (R-15: cite the datasheet, not the bench, for that one). **Unrolling, specialization, and jump-table dispatch** are measured trades (code size and fetch cost against loop overhead), with real crossovers; **self-modifying code** is a RAM-only, hot-loop-only, measured-win-only tool with loud warnings (it silently fails on ROM); **VDP throughput** is won by **amortization** (batch writes, set the address once and stream via `VMBW`'s auto-increment — never touch VRAM a byte at a time in a hot path); and **table-ism** — precompute everything, index at runtime — trades cheap memory for scarce time (the modern machine generates the tables, the 9900 looks them up). The **case study** tunes Chapter 17's scroller from 65% to 31% of frame, decision by measured decision in the doctrine's order, and the chapter closes on **when to stop**: when the frame fits, the next win is not worth it, and it is time to *ship* — the retro-dev virtue, and the only way the living platform got its games. The wait state is the cache miss's honest grandparent, and "placement is king" is the memory hierarchy's oldest and truest lesson, here measured to the cycle.
