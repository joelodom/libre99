## Ch. 1 — Genesis and Fall (drafted session 2; ~7,800 words ≈ 20 pp; status: draft complete, §1.9/Lab 1 re-founded session 3, pending review passes)


- The 4A merged two TI bloodlines: consumer silicon (calculators, Speak & Spell/LPC speech) and the TI-990 minicomputer line, whose single-chip TMS9900 (1976) was an early 16-bit micro with registers-in-memory workspaces.
- The cancelled 8-bit-bus TMS9985 forced the full 9900 onto an 8-bit multiplexed board: everything waits except 256 bytes of true-16-bit **scratchpad** — the book's central performance fact.
- GROM (TI-only serial-access ROM) + GPL (interpreted bytecode OS/software) implemented a closed cartridge market; it thinned the library, provoked the 1983 v2.2 lockout, and defines Part VI's subject matter.
- 99/4: June 1979, $1,150 with forced monitor, chiclet keys — a flop. 99/4A: June 1981, $525, real keyboard, 9918A adds bitmap mode.
- Price war vs. Tramiel's Commodore: $525→$99 (mid-1983)→$49.95 clearance; consoles sold below cost against a software annuity the closed box had already starved; reported Q2-1983 loss ≈$119 M.
- **Black Friday = Friday, Oct 28, 1983**: exit + reported ≈$330 M write-off; 99/8 cancelled; ~2.8 M total consoles, many sold *after* death in the strange $49.95 Christmas.
- Aftermath: orphan community — user groups, fairware, MICROpendium (1984–99), Millers Graphics (Explorer, GRAM Kracker), CorComp/Myarc, *TI Intern*, Geneve 9640 (1987), Chicago Faire annually since Nov 1983.
- Today's pillars (installed in Ch. 3): the **libre99 project** — the book's home and daily machine (pure-Rust console + `libre99asm`/`libre99gpl` toolchain + shipped original software incl. a clean-room firmware, Ch. 28) — with Classic99 (licensed TI software included), js99er, and MAME on the reference shelf; F18A, FinalGROM 99, TIPI, SAMS; xdt99 as the community toolchain. Ledger entries and style rulings R-1–R-7 logged this session.

## Ch. 2 — Grand Tour (drafted session 2; ~6,300 words ≈ 17 pp; status: draft complete, Lab 2 re-founded onto BENCH99 session 3, pending review passes)


- Cast (§2.1): TMS9900 CPU @ 3.0 MHz (registers = RAM workspaces); TMS9901 (CRU-driven I/O + interrupts + timer); TMS9918A VDP with private 16 K VRAM (frame interrupt, sprites; 9929A in PAL); TMS9919/SN94624 sound (SN76489 family) at `>8400`; console ROM 8 K (vectors, ISR, FP, **GPL interpreter**) on the 16-bit bus; 3 console GROMs (18 K GPL: OS, menu, TI BASIC) behind serial ports; scratchpad = **256 B at `>8300`–`>83FF`**, 16-bit, zero-wait.
- Fast domain = console ROM + pad only; **everything else crosses the 8↔16 multiplexer at a nominal +4 cycles/word** (measured precisely in Ch. 5). Three fabrics: memory bus, CRU, interrupts.
- Memory geography (§2.3): bare console CPU RAM = 256 B; the "16 K" is VRAM behind ports `>8800/>8802/>8C00/>8C02`; 32 K expansion fills `>2000` (8 K) + `>A000` (24 K), 8-bit; DSR window `>4000`; cart ROM `>6000`; GROM ports `>9800/>9802/>9C00/>9C02`; speech `>9000/>9400`. **Law: speed is a property of addresses, not instructions.**
- The tower (§2.4): BASIC (tokens in VRAM) over GPL (bytecode in GROM, VM state at `>83E0`/status `>837C`) over 9900 (interpreter in fast ROM). Book occupies Floor 1 (Part II+) and Floor 2 (Part VI); Floor 3 visited once (Ch. 28).
- Disorientation kit (§2.5): no OS-while-running, no scheduler, no protection, no heap, no file abstraction, **no hardware stack** (BL→R11; BLWP swaps workspaces); fully deterministic timing.
- **CQ-82** checklist named (§2.6, 9 items: response, attract mode, visual/sound discipline, speech + graceful degradation, QUIT etiquette, difficulty curve, zero crashes, packaging) — Part IX's rubric.
- Lab 2: same `JMP $` stepped at `>8300` vs `>A000` on BENCH99 (bare bench: `pw`/`pc`/`s`), measured 10 vs 14 cycles — the funnel's +4, machine-verified; Part A traces the live GPL interpreter in console ROM (`boot`, `s 12`). Road-map table (§2.7) binds every book part to a block-diagram territory. New style ruling R-8 (named recurring artifacts get ledger handles).

## Ch. 4 — The TMS9900 (drafted session 2; ~7,100 words ≈ 23 pp; status: draft complete; Lab 4 re-founded onto BENCH99 session 3 (trace.a99 → cartridge + `WS EQU >8300`); all listings machine-verified via libre99asm at commit 5b1c0d0; MOV dest-pre-read emulator deviation ledgered; remaining audit: datasheet cycle counts C/M for MOV/JMP against TI manual during App. A drafting)


- Programmer-visible state = **PC, WP, ST — nothing else**; R0–R15 are RAM at **WP + 2n** (`LWPI`/`STWP`, `LIMI`/`STST`; PC read only via `BL` idiom). Interrupt vectors `>0000`–`>003F`, XOP vectors `>0040`–`>007F` (WP,PC pairs).
- Reserved seats: R0 shift-count / **never an index**; R11 `BL` link (+ XOP source EA); R12 CRU base; R13–R15 context linkage (old WP/PC/ST land in the **new** workspace; `RTWP` inverts; callee reads caller's Rn at `@2n(R13)`). Book convention declared: **R10 = software stack pointer** (built Ch. 9).
- Consequences (§4.3): context switch = pointer swap (`BLWP`/interrupt/`XOP` are one mechanism); **registers have addresses** (self-referential idioms live); **every register touch is a memory access** — incl. the dest-read-before-write habit (MMIO implications owed to Ch. 12).
- Memory model (§4.4): 32 K aligned words, A15 ignored on word ops (silent!), **big-endian**; **byte ops on registers use the HIGH byte** — constants ride high (`>4100`, `'A'*256` `[libre99asm]`), `SWPB` rescues (and sets no flags), byte writes preserve low bytes, `*Rn+` steps by operand size.
- ST (§4.5): L>/A>/EQ/C/OV/OP/X + mask (ST12–15). Compares grade **S vs D**; everything else grades **result vs 0** (⇒ `MOV` is a free test). Jump grammar: H/L unsigned, GT/LT signed; **no single-op signed ≥/≤** (inversion or pair idioms); C = no-borrow after subtract; `DEC`/`JNE` loops as in HELLO.
- No stack (§4.6): `BL`→R11 depth **one** (`RT` = `B *R11`); non-leaf saves R11 first; `BLWP` = heavy modular call; stacks are Ch. 9 equipment, not ambient.
- Formats (§4.7): five addressing modes (R, `*R`, `*R+`, `@sym`, `@sym(Rn≠0)`) on **both** operands of Format I = memory-to-memory philosophy; jumps reach ±~127 words; immediates are the exception, not the idiom; full encodings → App. A.
- Timing (§4.8): **T = C + 4 × (accesses in 8-bit domain)**; `MOV R,R` = 14c/4a → 14/18/30 by placement (datasheet truth; our emulator's missing MOV dest-pre-read makes it measure 26 in the all-slow row — deviation ledgered for Ch. 5); `JMP` = 10c/1a (matches Lab 2's bench numbers); fetches always count; 50,000-cycle frames scale intuition.
- Peers (§4.9): alone at 16 bits in 1976; per-op slow but per-op big; undone by 64 K + 64 pins + insularity; 6502 zero page = the workspace idea inverted. Lab 4: paper-trace T1–T6 (incl. R11=HALT, flag persistence, `MOVB` high-byte verdict), graded on BENCH99 (`u`/`s 6`/`r`) with registers watched *in memory* (`m 8300 32` — WS is an `EQU >8300` now that the program is a cartridge); Part C reseats WP mid-flight (`wp 8308`). New ruling R-11 (exercises may cite App-A premises).

## Ch. 5 — The Console Memory Map (drafted session 4; ~6,000 words + code ≈ 20 pp; status: draft complete; all §5.1–5.7 + Lab figures MACHINE-VERIFIED on BENCH99 at commit 2da67ae — race table, mirror set, padrun speedup, MMIO side-effect, det32k; MOV dest-pre-read deviation reconfirmed OPEN and surfaced to the project as a cpu.rs fix; code in code/ch05/)


- The full map, south to north: `>0000` console ROM (8 K, 16-bit — vectors, GPL interpreter, ISR, FP package) · `>2000` low expansion (8 K) · `>4000` DSR window · `>6000` cartridge ROM (8 K) · `>8000` control block (scratchpad + ports) · `>A000` high expansion (24 K). Ports: sound `>8400` (w); VDP `>8800`/`>8802` rd, `>8C00`/`>8C02` wr; speech `>9000` rd / `>9400` wr; GROM `>9800`/`>9802` rd, `>9C00`/`>9C02` wr — all side-effectful.
- Two domains: the fast island (console ROM + scratchpad, 16-bit, zero-wait) versus everything behind the funnel; §5.3's rig measured the toll as a flat **+4 per word crossing**, identical at `>2000`, `>6000`, and `>A000`, upgrading the "≈ +4 nominal" ledger row to measured; the law stands — speed is a property of addresses, not instructions.
- Fetch and operand tolls are separate line items: `JMP $` = 10 pad / 14 slow; `MOV R0,R1` = 14 all-pad / 18 code-slow / 26 all-slow on our core against the datasheet's **30** — the MOV destination pre-read deviation, still open at commit `2da67ae`; `A R0,R1` models the pre-read and measures the correct 14 / 18 / **30**, so it is the honest probe where that toll matters; the `cpu.rs` fix was surfaced to the project, not written from the book.
- Scratchpad `>8300`–`>83FF`: the machine's only 16-bit RAM — and an unexpanded console's only CPU RAM at all; first coarse tenant map given (GPL WS `>83E0`, GPL status `>837C`, ISR/GPL churn in `>83C0`–`>83FF`, quiet claimable corner near `>8320`–`>836F`); byte-level atlas deferred to Ch. 24 / App. C.
- Partial decoding: the pad mirrors four times across `>8000`/`>8100`/`>8200`/`>8300` (the decoder ignores A8–A9), and a write through a mirror lands on the live pad; unowned space never faults; reading ports has side effects (two reads of GROM `>9802` returned `>60` then `>01`) — reading is not looking.
- The 32 K arrives split 8 + 24 because `>4000`/`>6000`/`>8000` were already spoken for; the hot-loop-into-pad trick was demonstrated and measured (padrun.a99): 1000 laps fell from **28,062 to 20,058 cycles**, a **1.40× speedup**, by moving the loop's bytes onto the island.
- §5.7 established the memory-budget worksheet (blank form: `code/ch05/budget-template.md`) — every later project fills it first; the VRAM column is reserved until Ch. 12.
- Artifacts in `code/ch05/`: rig.a99 + race.txt (the timing rig, Lab 5), padrun.a99, mmio.a99, det32k.a99, budget-template.md.

## Ch. 7 — Instruction Set I: Moving Data (drafted session 4; ~9,000 words + code ≈ 26 pp; status: draft complete; all listings assemble via sh verify.sh, every §7.1–7.6 + Lab figure MACHINE-VERIFIED on BENCH99 at commit 2da67ae; memlib self-tests green/sabotage red; libre99asm operator-precedence (left-to-right) and R0-index rejection pinned; code in code/ch07/)


- Five general addressing modes serve both operands: `Rn`, `*Rn`, `*Rn+` (steps by operand size — the loop mode), `@LABEL`, `@LABEL(Rn)`; R0 cannot index (libre99asm rejects `@X(R0)`); immediates are a separate family — `LI`/`LWPI` load, `STWP`/`STST` introspect, `CLR`/`SETO` are the free constants, `SWPB` the byte wrench. Mode encoding/cost table established here feeds App. A.
- The high-byte law is big-endianness seen through Rn = WP + 2n: a register's byte is the byte at its own even address — the high half. Byte cargo rides the high half book-wide (now a calling convention); `SWPB` repairs; word ops silently align (A15 ignored); `MOVB` is the instruction that talks to the ports, so the law governs all future device I/O.
- `MOV`/`MOVB`/`LI` test their cargo (flag table §7.2, bench-verified): a move is a measurement; `CLR`/`SETO`/`SWPB`/`STWP`/`STST` set no flags (a surprise for CLR/SETO); `MOV R1,R1` foreshadowed for Ch. 8.
- The pump is `MOV *R1+,*R2+`; the family is copy/fill/compare/scan (plus the nailed-destination port feeder, promised to Ch. 12); in the basic loop one instruction moves and two administrate, which unrolling amortizes. Same-register autoincrement pinned on the bench: source fetched entirely (incl. its increment) before the destination resolves.
- `@X(Rn)` wears two costumes — array (label base, register index, ×2 for word tables) and record (`EQU` offset, register base). Words for computing, bytes for storing and shipping. **libre99asm has no operator precedence (left-to-right)** — precompute mixed expressions.
- Timing (measured): addressing-mode add-ons `*Rn` +4, `*Rn+` +8, `@` +8 plus the fetched address word; the pump = 30 island / 42 buffers-expansion (hw) / 46 all-expansion; the copy loop 50 / 62 / 74 cycles per word → **≈ 1000 / 806 / 675 words per frame**; unrolling ×2 → 40/word (1250 wpf). Expansion-destination pump/fill/copy inherit the Ch. 5 MOV dest-pre-read deviation (core 4/unit light; A-family cross-probe confirms the hardware figure).
- `memlib` born — `MEMCPY`/`MEMCPB`/`MEMFIL`/`MEMCMP`/`MEMSCN`, `BL`/`RT` leaf linkage, args R0–R2, byte cargo high half — self-tested by border-color verdict (green `>02` pass / red `>06` fail) with the sabotage drill (pattern seeds SYSCHK, Ch. 11); measured with Ch. 5's discipline; seeds `lib99` (Ch. 11).
- Border verdict recipe (two writes to `>8C02`: color byte then `>87`) borrowed on faith; anatomy owed to Ch. 12; `GREEN`/`RED` become book-wide convention.

## Ch. 8 — Instruction Set II: Arithmetic, Logic, and Bits (drafted session 4; ~9,700 words + code ≈ 24 pp; status: draft complete; all listings assemble via sh verify.sh, every §8.1–8.8 flag/cycle/layout fact MACHINE-VERIFIED on BENCH99 at commit 1ae787c; mathlib self-tests green (>C0DE/>0000/>0000, LFSR period counted to >FFFF) / sabotaged-tap red (id 29); DIV flat-cost timing simplification surfaced to the project; code in code/ch08/)


- Ch. 8 completes the computational instruction set. Flag semantics: every arithmetic op files a
  perishable report in ST; A/AB/S/SB/AI/INC/INCT/DEC/DECT/NEG/ABS all set the full L>/A>/EQ/C/OV
  (byte forms add OP; word forms never touch OP). INCT/DECT write C — the §8.6 pitfall; NEG/ABS of
  >8000 return >8000 with OV set; ABS's flags read the original operand. Flag-silent family (extends
  Ch. 7's SWPB): SWPB, CLR, SETO, STWP, STST, LWPI, LIMI, B, BL, BLWP, MPY, + CRU/external — nothing
  you want mid-calculation.
- Comparison C/CB/CI (first vs second operand) non-destructive; two rulers — JH/JHE/JL/JLE unsigned,
  JGT/JLT signed, no single-op signed ≥/≤ (pair idiom); >8000 vs >0001 the canonical disagreement
  (JH takes, JGT does not — bench-shown). COC/CZC report through EQ alone.
- Booleans: ANDI/ORI immediate-to-register; SOC = OR, SZC = AND-NOT reaching memory (+B forms);
  XOR/INV want a register. Packed-field idioms (test/set/clear/toggle/extract/insert) in
  code/ch08/fields.a99.
- Shifts SLA/SRA/SRL/SRC: count 0 → R0's low nibble, 0-there → 16; carry = last bit out (a data
  channel); SRA rounds toward −∞ (−5→−3, not −2). Cost slope measured 2 cyc/bit (12+2n island;
  R0-count path +8). MPY/DIV register-PAIR: high word in the lower register; MPY Rd:Rd+1 = product,
  DIV Rd = quotient / Rd+1 = remainder. Costs (island/cart): MPY 52/56, DIV 92/96 (flat on core;
  datasheet data-dependent — noted), DIV-overflow abort 16/20. ×10 showdown: shift-and-add 76 beats
  MPY 94 for a small constant. DIV overflow iff high word ≥ divisor; on refusal OV set, dividend
  pair intact, fast bail — DIVSAF guards it. MULS/DIVS signed on MPY/DIV; remainder takes the
  dividend's sign (book-wide).
- 32-bit module ADD32/SUB32/CMP32/NEG32 — pair convention: high word in the lower register
  (book-wide, matches MPY/DIV). No ADC: carry chains need flag-silent interleaving only. S borrow
  sense: no-borrow SETS C, borrow CLEARS C (bench-confirmed). INCT-in-carry-chain pitfall
  (carrybug.a99): buggy sum >00000000 vs fixed >00010000. U16DEC (divide chain, flat 1234 cyc) /
  U16DEB (subtract table, 1524–2978 cyc, forward + leading-zero policy) / HEX16; digit = value + >30.
- RND: 16-bit maximal LFSR (Galois), tap >B400 (x^16+x^14+x^13+x^11+1), period 65,535 PROVEN by
  bench count (>83F8=>FFFF); zero stays zero, RNDS refuses it; RNDB ranges via MPY high word. Console
  seed byte observed at >83C0 (moves 0000→A9BE→08C2 across frames; TI Intern maps the seed here,
  hedged; ticks only while the ISR runs — bare carts seed from the player, Ch. 21).
- Lab 8 = the book's first self-checking lab: routines + harness ship in one file (mathlib.a99, the
  memlib precedent) with code/ch08/mathtest.bench; verdict at >83FA/>83FC/>83FE = done/FAILS/
  first-fail, PASS = >C0DE/>0000/>0000 (29 vectors, machine-verified); a sabotaged tap turns it red
  (>03FF/FAILS 1/id 29). Harness pattern feeds lib99 (Ch. 11) and the CQ-82 discipline (Part IX).

## Ch. 9 — Control Flow and Program Shape (drafted session 5; ~7,900 words + code ≈ 26 pp; status: draft complete; all six listings assemble via sh verify.sh, every §9.1–9.9 + Field Notes + Lab fact MACHINE-VERIFIED on BENCH99 at commit b9291be; verdict files green (reach/link/stack99/task99 → VDP R7=>02), task99 log interleaves AA BB AA BB AA BB AA BB, stack99 FACT(8)=>9D80 with R10 balanced to >8380, XOP transfer observed into console vector {>280A,>0C1C}; new ruling R-16; code in code/ch09/)
- Control flow = jumps + `B` + `BL`. A conditional jump reaches ±128 words (the assembler quotes
  −254..+256 bytes when exceeded); the two-jump idiom (invert the condition, jump over a `B @far`)
  branches long since `B` reaches all 64K. Costs (cart code, pad WS): JMP 14, Jcc 14/12, B @L 24,
  B *R 16, BL 28, RT(=B *R11=>045B) 16, BLWP 50, RTWP 18, X 38, dispatch MOV 34.
- `BL` links the return address in **R11** — one register. A non-leaf MUST save R11 before any inner
  call and restore before RT, or it returns into itself (the one-breadcrumb bug; link.a99 shows
  before/after R11 differing). A leaf keeps R11 and RTs directly. This is the chapter's opening
  anxiety (the Dayton vignette) and its first discipline.
- **R10 = the book's software stack** (full-descending): PUSH = DECT R10 / MOV x,*R10; POP =
  MOV *R10+,x. Recursion proves composition: stack99 computes FACT(8)=40320=>9D80 and returns R10
  balanced to >8380. **Ruling R-16** fixes the book-wide calling convention (args R0–R2 caller-saved;
  leaf keeps R11; non-leaf saves it; R10/R13–R15 sacred; errors in R0, 0=success) — every lib99
  routine (Ch. 11+) obeys it.
- **BLWP @vec** (50c) = the context-switch call: a 2-word vector {WP,PC} placeable anywhere in RAM
  becomes the new context, and old WP/PC/ST are saved into the new R13/R14/R15; **RTWP** (18c)
  restores them. Verified linkage in probe.a99: >8300/>6040/>C000. Use when a routine wants its own
  16 registers (interrupts Ch. 22, DSRs Ch. 30, coroutines); a BL suffices otherwise.
- Structured shapes are named jump patterns (if/else, while, repeat, for-down-to-zero). A **jump
  table** — MOV @TAB(Rn),Rx (selector×2) + B *Rx — makes an n-way switch two instructions (selector
  1 → case D1 → >00D1). **X** executes an instruction fetched elsewhere (X @XTGT ran INC R5, 38c).
  **XOP** traps ROM vectors at >0040+n×4 (you can't install your own on a stock 4A; new R11 = operand
  effective address; ST X-bit set; XOP-0 vector reads {>280A,>0C1C}) — heritage (990/Geneve), not a
  daily 4A tool.
- **Coroutines = workspace switches** (each owns a workspace, so its state survives a yield free).
  Lab `task99` is a cooperative multitasker: two tasks, yield = BLWP @YVEC to a scheduler that
  saves/rotates/restores 3-word {WP,PC,ST} control blocks; the shared log interleaves
  AA BB AA BB AA BB AA BB (fair switch proven), green, in 160 bytes (≈50-byte scheduler core; 216 B
  whole file). Error handling is convention, not machinery: flags for the immediate caller, an R0
  code for errors that must travel, the R10 stack for safe manual unwinding.
- Field Notes read the console's own vectors cold: reset >0000={>83E0,>0024}, level-1 VDP interrupt
  >0004={>83C0,>0900} (ISR workspace >83C0 = the Ch. 8 seed-byte corner), XOP block >0040+ — the
  machine's reflexes ARE the BLWP-and-a-table the chapter just built. Seeds forward: R10 stack + R-16
  → lib99 (Ch. 11); BLWP → interrupts (Ch. 22) & DSRs (Ch. 30); coroutines/state machines → game
  loops (Part IX); jump tables → command interpreters & GPL dispatch (Ch. 26).

## Ch. 10 — The CRU: TI's Serial Nervous System (drafted session 5; ~5,500 words + code ≈ 14 pp; status: draft complete; all three listings assemble via sh verify.sh, §10.1–10.6 + Lab machine-verified on BENCH99 at commit 84c585a; keyscan/crudemo/cruscan green; code in code/ch10/)
- The CRU (Communications Register Unit) is a 4,096-bit space of single-bit I/O, entirely separate
  from memory, worked by five instructions — the home of every non-memory-mapped device (keyboard,
  joysticks, cassette, interrupt mask, expansion cards). Heritage: the 990 minicomputer's
  industrial-control bit bus. Vignette = the cassette-relay CLICK (single-bit output made physical).
- R12 holds the CRU base, and the value in R12 is 2× the hardware bit number (address in R12 bits
  3–14): bit >0012 ↔ R12 >0024, disk-card bit >0880 ↔ R12 >1100. Verified. Single-bit ops take a
  signed bit displacement from the base.
- SBO/SBZ set/clear a bit (outputs, flag-silent); TB tests a bit into EQ (input, the only ST-toucher);

## Ch. 12 — Inside the TMS9918A (drafted session 6; ~6,200 words + code ≈ 18 pp; status: draft complete; vdplib.a99 assembles via sh verify.sh, §12.1–12.7 + Lab machine-verified on BENCH99 at commit e97e8ce; self-test green + the new independent `vram` oracle confirms VRAM; code in code/ch12/)
- The VDP (TMS9918A) is a SEPARATE processor with 16 KiB private VRAM the CPU cannot address —
  reached only through 4 ports. NO framebuffer: the picture is a continuous per-scanline process
  built from tables and forgotten. Vignette = the missing framebuffer (an Apple II porter finds no
  screen memory anywhere in the map). Bridge for a 2026 CS reader: coprocessor-with-own-memory (GPU) +
  tilemap, minus the framebuffer.
- Table-driven is a NECESSITY, not a quirk: a 256×192 4-bpp framebuffer = 24 KiB (exceeds VRAM) and
  would take ~40 frames to fill through the slow ports; the three tables describe a full screen in
  <3 KiB and redraw it in ~1 frame. (The book's compression/bandwidth argument.)
- Four ports: >8800 read data (prefetched, auto-inc), >8802 read status (clears F/5S/C + resets addr
  flip-flop), >8C00 write data (auto-inc), >8C02 write addr/register. Two-byte handshake: low 8 bits,
  then 00=read / 01=write / 1xxxx=register; auto-increment ⇒ aim once, then stream. Written longhand
  once, then vdplib forever.
- Registers R0–R7 are WRITE-ONLY (keep a RAM shadow): mode bits M1/M2/M3; R1 master switches (BL
  blank/enable, IE interrupt-enable, sprite size/magnify); table-base pointers R2–R6 (coarse
  granularity, movable in one write); R7 = fg(hi)|backdrop(lo) — the border-verdict color is a VDP
  register write all along. Status F/5S/C = frame heartbeat; reading CLEARS the flags → one owner reads
  it per frame (Ch. 17's loop exists for this).
- The three tables (name→pattern→color, per cell per frame) ARE the whole of Part III; each later mode
  is a different table layout. Timing (bench-measured, e97e8ce): addr-port write 34c, VRAM byte
  ~40–44c, bulk copy ~68–72c/byte, name-table clear ~1 frame — the numbers that make the table model
  provably necessary. (VDP-port MOVB costs inherit the MOV dest-pre-read deviation.)
- Lab vdplib (lib99): VWTR / VWA / VRA / VSBW / VMBW / VSBR / VMBR — the port protocol as named ops
  (R-16), verified green THREE ways (own read path, self-comparison, independent `vram` oracle). Cross-

## Ch. 13 — Graphics I Mode and a Real Text Engine (drafted session 6; ~5,350 words + code ≈ 15 pp; status: draft complete — concise vs the 24pp target, an expansion candidate in review; textlib + MONITOR99 machine-verified on BENCH99 at commit 31417ef; screen shows the output, dump matches memory; code in code/ch13/ + vdplib refactored to .inc)
- The machine has no `print`. Graphics I = three tables: name (768 B, one pattern# per 32×24 cell),
  pattern (256 × 8-byte glyphs = the font), color (32 B, one fg/bg pair per GROUP of 8 pattern
  numbers). Vignette = the word you cannot print, then cannot color (the eight-character rule).
- The eight-character color rule: a char's color = color-table[pattern# ÷ 8]; chars share a color iff
  their pattern numbers fall in the same group of 8. Concrete: "HELLO" (ASCII >48/45/4C/4C/4F) spans
  groups 8–9 = ASCII >40–4F, so you can't recolor it alone. A 32-vs-768-byte compression trade you
  plan around by giving same-colored things pattern#s in one group. THE constraint that defines the
  TI look.
- textlib (lib99) = the print statement: CURPOS (cursor as 0–767 cell offset), TXPUTC/TXPUTS (char/
  string, CR + wrap), TXLOC/TXCLS/TXMODE, TXHEX (numbers). Convention: name-table byte = the char's
  ASCII code (so `screen` shows text and pattern[ASCII] is the glyph). Verified green; screen shows
  C0DE/DEAD.
- Composition: a lib99 module used by another module must be a routines-only .inc + a .a99 self-test
  harness (COPY deps in dependency order — flat namespace, no linker). vdplib refactored to
  vdplib.inc + vdplib.a99 here; textlib.inc is the first module to consume another. README updated
  (extends R-17).
- Two textlib bugs found in verification (keep as teaching): TXMODE wrote the enable byte to reg 0 not
  reg 1 (display blanked — `screen` blind, `vdp` caught it); VSBR returns the byte in R1 HIGH (compare
  CB not CI). Fonts: ship your own (textlib's hex digits) or borrow the console's small-caps font from
  GROM (Ch. 25); no true lowercase built in. Numbers-on-screen = the debugging dashboard (Ch. 11 kin).
- Color strategy: uniform / highlight-by-group / inverse-video (swap nibbles) / palette-by-group.
  Screen composition: borders, panels, reserved status line (row 23, book-wide). Double buffering
  (§13.7): two name tables, draw the hidden one, flip R2 — atomic/tear-free (Ch. 17 uses it). 32
  columns because the VDP drives a TV (256 px fits inside overscan).

## Ch. 14 — Text Mode and Multicolor Mode (drafted session 7; ~5,800 words + code ≈ 14 pp; status: draft complete; textlib40 + mcolib + carousel + monitor40 machine-verified on BENCH99 at commit 0d3e5d5; the new `pixels` framebuffer oracle renders the chip's actual output; code in code/ch14/)
- The two modes flanking Graphics I. A **mode** is a contract with the beam about how to read the same 16 KiB of VRAM; four modes select on M1/M2/M3 (R1 bit4 / R1 bit3 / R0 bit1). Text leans toward words; multicolor toward a painted canvas. Neither is the destination (bitmap, Ch. 15) — the transferable skill is switching between them.
- **Text mode** (M1): 40×24, 6-pixel cells (the chip draws only the TOP SIX bits of each pattern byte), ONE fg/bg pair in R7 (hi/lo nibble) — NO colour table, NO sprites. Setup R1=>D0 (16K + enable + M1), R2=>00 name, R4=>01 pattern; R3/R5/R6 unused. A font drawn 5 wide (bits 7–3) renders IDENTICALLY in Graphics I and text → one font, both modes (verified glyph-by-glyph with `pixels 1`).
- **textlib40** (lib99): TX40MD/TX40CL/TX40LC/TX40PC/TX40PS/TX40HX, cursor CUR40=>8346. Two wrinkles from 40 not being 2^n: positioning does `row*40 = (row<<5)+(row<<3)` (shift-pair, no MPY), and carriage return needs a `DIV` (no mask finds a row start). Powers of two are free; everything else costs an instruction. Foundation for the Ch. 42 editor; screen width ≠ document width (TI-Writer windows a wider logical line — Field Notes).
- **Multicolor mode** (M2): 64×48 fat 4×4 pixels, each any colour or transparent. Same 32×24=768 name table but bytes read as colour PAIRS; within each 4-row group, rows read definition bytes 0–1/2–3/4–5/6–7. Fill the name table with map `name[i]=((i>>7)<<5)|(i&31)` so 768 entries spread over 192 defs = a flat 1536-byte canvas. **PLOT64** address = `(y>>3)*256 + (x>>1)*8 + (y&7)`, high nibble for even x (left block), low for odd. mcolib = MCMODE/MCCLS/MCPLOT (RMW one nibble). Colour 0 = TRANSPARENT everywhere (a hole to the backdrop, not a colour). A museum piece — coarse, and the bitmap does real graphics better; recognize it, know why you skip it.
- **Clean mode switching** = four movements: BLANK (clear R1 bit6), rewrite mode + base registers, re-lay-out the new mode's VRAM, UNBLANK. Blank first because the beam draws your half-built screen otherwise. Three standing hazards: forgetting to blank (garbage flash, invisible under single-step), forgetting to rebuild the tables (old bytes in the new font), and clobbering the 16 KiB bit (R1 bit7) — why BLANK writes >80 not >00. Place shared tables at the same addresses (name >0000, pattern >0800) so a Graphics I↔text switch moves no font bytes: a mode's tables are a lease, not a deed.
- Lab: **carousel** (Graphics I 32 colour bars via the 8-char group rule → text via textlib40 → multicolor X-diagonal via mcolib; all three phases verified switching cleanly), **PLOT64** (16 colour bars exact via `pixels 4`, byte-level self-check), **monitor40** (MONITOR99 at 40 columns, 6 words/row, dump matches `m 0000` to the byte). BENCH99 gained `pixels [step]` (renders the real 256×192 frame, one palette-index hex digit per sampled pixel — Part III's pixel oracle) and a mode-aware `screen` (40 wide in text). Seeds: bitmap (Ch. 15), sprites over a background (Ch. 16), a frame-timed carousel (Ch. 17), the editor (Ch. 42).

## Ch. 15 — Bitmap Mode: Graphics II (drafted session 7; ~5,800 words + code ≈ 14 pp; status: draft complete — concise vs the 26pp target, an expansion candidate (span-fill impl, more clash examples, BSTAMP) in review; bmplib + grapher machine-verified on BENCH99 at commit 0d3e5d5; code in code/ch15/)
- Bitmap mode (Graphics II, M3=1) = a bit per pixel over 256x192, the closest the 9918A comes to a framebuffer — but "a framebuffer with three rules": a BIT not a colour per pixel (6 KiB), colour shared per 8x1 strip (a second 6 KiB — the attribute clash), and no cheap update (12 KiB in VRAM, one byte at a time through the port). Vignette = the 12 KiB single picture that fills like a fax.
- Screen in THIRDS: each third has its own 2 KiB pattern + 2 KiB colour; the 768 name entries, filled 0..255 x3, unroll into a linear 6 KiB pixel canvas + parallel 6 KiB colour table. The 8x1 colour-pair rule (the clash): one fg/bg per 8 horizontal pixels; align colour to the 8-grid, master not mourn.
- Setup = 6 register writes; the notorious mask bits: **R4=>03** (not >00, which cramps the pattern to a repeating 2 KiB — the classic bug), **R3=>FF** (not >7F, which drops colour onto the pattern). R3/R4 are select-bit + address-mask, the chip's raw address lines showing through.
- Addressing: pixel (x,y) -> pattern byte `(y AND >F8)*32 + (x AND >F8) + (y AND 7)`, bit `>80>>(x AND 7)`, colour twin at +>2000. bmplib: BMODE/BCLS/BPSET (RMW, bit mask from an 8-entry lookup — variable shift needs R0)/BLINE (Bresenham, integer-only, signed compares because err goes negative). Verified: box + X, all edges exact (vram >17FF=>FF, >0CF8=>01); grapher plots a parabola with MPY.
- Text over bitmap = OR font glyphs into 8 consecutive pattern bytes (cell-aligned); fills = byte-aligned >FF writes in the middle, pixels at the ragged ends, colouring strips as you go. Full-screen images = a PC-side clash-aware DITHERED conversion (Ch. 38 tooling, no python on the PC box) ending in one bulk VMBW; fades = colour-table animation.
- Cost ledger: 12.75/16 KiB committed, no room to double-buffer. Measured truth (§15.8): bringing up + clearing the bitmap = **869,894 cyc ≈ 17 frames**, so full-screen animation is impossible — touch few pixels (dirty-rectangle), the discipline Ch. 17 builds the game loop around. Sidebar: the plain-9918 (no A) owners couldn't run this — capability tiers (Ch. 44). Seeds: sprites over any background (Ch. 16), the beam + frame budget (Ch. 17), the asset pipeline (Ch. 38).

## Ch. 16 — Sprites (drafted session 7; ~4,800 words + code ≈ 18 pp; concise vs the 26pp target; status: draft complete; spritelib + DODGE machine-verified on BENCH99 at commit 0d3e5d5; code in code/ch16/)
- Sprites = the VDP's 32 hardware-composited moving objects, drawn OVER the background with no redraw — move by writing 2 bytes (Y,X), the beam composites for free. This is what made the 4A a games machine. The gift comes with limits, and the limits are the lesson. Vignette = the object that floats (change X, background never smears).
- Sprite Attribute List (R5, spritelib at >0300): 32 x {Y, X, pattern#, early-clock|colour}. Y = the line ABOVE (SPUT writes screen-y-1); Y=>D0 ENDS the list (forget it -> phantom sprites); early-clock bit = 32 px left (smooth left entry); one colour/sprite. Size 8x8/16x16 + 2x magnify = global R1 bits. Shapes = 8-byte bitmaps in the R6 pattern table (spritelib >1000).
- **Law of four**: max 4 sprites/scanline; the 5th sets the fifth-sprite flag + is DROPPED (lower# wins). Verified: 5 balls on a row -> pixels shows 4, the 5th (x=216) gone. **Flicker multiplexing** rotates which 4 draw each frame to fake more (the asteroid-belt shimmer). Field Notes: count sprites frame-by-frame to read the trick; Parsec's belt reserved for the author.
- Motion: SMOV rewrites Y,X. Smooth sub-pixel = **8.8 fixed-point** (whole pixel hi byte, 256ths lo), add velocity/frame, hand the hardware only the hi byte. Console firmware auto-motion (VDP >0780 table, frame ISR) is FREE but constant-velocity only — serious games bypass it for anything that accelerates/bounces/steers.
- Collision: hardware = ONE bit (coincidence: some pair touched, not which/where) -> games use **bounding boxes** (|dx|<8 && |dy|<8), verified firing in DODGE (DHITS=1). Hybrid = flag as cheap early-out gating the box tests. Lower sprite# = priority; colour 0 = transparent (shapes sprites, lets them layer). Animation = change pattern# over time (frame tables / ping-pong / state-driven), one byte, nearly free.
- Lab: spritelib (SINIT/SPAT/SPUT/SMOV/SHIDE) + **DODGE**, our first game engine — player + 4 falling meteors, respawn, bounding-box collision, score/lives — deterministically proved on the bench. DODGE grows a real loop in Ch. 17, sound in Ch. 19, playable input in Ch. 21. Seeds: the 60 Hz loop (Ch. 17), input (Ch. 21), the ISR (Ch. 22).

## Ch. 17 — Motion: Game Loops, Scrolling, and the 60 Hz Contract (drafted session 7; ~4,800 words + code ≈ 18 pp; concise vs the 26pp target; status: draft complete; terrain + smooth machine-verified on BENCH99 at commit 0d3e5d5; code in code/ch17/)
- The frame is the machine's heartbeat AND your budget: the VDP raises the F flag (status bit7, cleared by reading status) at the end of each frame; the beat interval is a MEASURED **~50,000 cycles** (50,014/frame, 50,011.7 avg over 10) — ~1,200 VDP writes for EVERYTHING. Vignette = sixty times a second, no matter what. Bridge: no delta-time, no vsync compositor, no GPU, no spare cycles — a fixed beat you fit inside.
- Two loop shapes: **poll-synchronized** (int off, do work then spin on F — full control, no net) and **interrupt-driven** (int on, the beat calls a handler — free background work but interrupts land mid-code). Separate UPDATE (CPU-RAM, safe anytime) from RENDER (VRAM writes batched into the vertical-blank SAFE WINDOW after F); render only what changed (dirty rectangles). Bare bench can't drive F (no beam per instruction) — loops verified by measurement, run under the beam in Ch. 22.
- Scrolling exposes the 9918A's biggest limit: **NO hardware scroll register**. Full-screen coarse scroll = rewrite 768 name cells = **~163,000 cyc ≈ 3.3 frames** (measured, terrain.a99) — does NOT fit. Escapes: scroll a band, scroll less often, or...
- **Pattern-shift smooth scroll** (the Parsec method, §17.5): leave the name table; redraw the band's few "scroll" characters' PATTERNS shifted 1 px/frame (feed the map's next column), one coarse band-shift every 8th. Measured **~1,044 cyc** (smooth.a99, pattern byte walks >80→>40→>20) vs ~163,000 — ~150x cheaper, inside the frame. Pays for #tiles not #cells. THE technique.
- Vertical scroll = the same rotated; split scroll = change a base register mid-frame (Ch. 18); parallax = bands on different clocks; colour cycling = flow with no pixel moved. Page flipping (Graphics I two-name-table R2 flip, Ch. 13) = tear-free motion where VRAM allows (not bitmap). Time system for lib99 = a frame counter + countdown timers = scheduled events.
- Pitfalls: the **VDP address-register race** — an interrupt landing inside the two-byte counter aim sends the main loop's 2nd byte to the wrong address (the classic unreproducible crash); fix = mask interrupts (LIMI) around every VDP access. Lab: terrain (coarse + budget) + smooth (pattern-shift); the frame-budget HUD + full TERRAIN flyover assemble in Ch. 22 (profiling timer). Seeds: mid-frame tricks (Ch. 18), the ISR (Ch. 22), the capstones (Part IX).

## Ch. 19 — The Sound Generator: Music and Effects Engineering (drafted session 7; ~4,500 words + code ≈ 18 pp; concise vs the 26pp target; status: draft complete; sndlib machine-verified on BENCH99 at commit 74c9b90; code in code/ch19/)
- The TI's sound is the **SN76489**: 3 square-wave tone channels + 1 noise, commanded one byte at a time through the write-only port >8400. No samples — you SYNTHESIZE sound from arithmetic (a zap is a falling pitch you write; an explosion is decaying noise). Vignette = four voices, one byte at a time. BENCH99 gained a `sound` oracle (the chip is write-only) — every claim verified.
- Protocol: LATCH byte (bit7=1: channel cc, type t, low 4 data) / DATA byte (bit7=0: high 6 bits of the last-latched tone period). Tone period = 2 bytes, attenuation = 1. Writes only land at EVEN addresses (wsndbyte). sndlib STONE writes the sequence.
- Frequency: **f = 3,579,545/(32*N)**, N 10-bit -> ~5 octaves (~109-3494 Hz). A4 -> N=254 (verified 440.4 Hz). Equal-tempered NOTES table (C4=428..B4=226, within +/-0.25%); octave up = N>>1. Verified via a C-major chord: C4 261.4/E4 330.0/G4 392.5 Hz.
- Attenuation 4-bit (0 loud, 15 silent, 2 dB/step); ENVELOPES = attenuation tables stepped per frame (pluck/swell/decay). Noise (ch3) = 15-bit LFSR: white vs periodic, 3 rates or follow-tone-2 (tunable); drums/engines/explosions = noise + rate + decay. sndlib SVOL/SNOISE/SSILNC.
- Effects (sfxlib) = register sweeps per frame: pitch slide (zap/power-up), vibrato, ARPEGGIO (the poor chip's chord — one channel cycling a triad fast), alarms, sirens. Console SOUND-LIST format = duration-tagged register streams, one format the whole platform shares (firmware/GPL/asm), played unattended by the ISR auto-player (Ch. 22). Music driver = tracker patterns/orders + CHANNEL-STEALING (effects steal a music voice, then restore it).
- Modern pipeline: tracker -> VGM (logged chip commands) -> TI sound-list/pattern data (the converter is a Ch. 38 asset script). Lab: sndlib + scoring DODGE (title tune + loop + 6 effects, channel-stealing) — the score assembles across Ch. 17 (loop) + Ch. 22 (auto-player). Field Notes: the Space Invaders descending loop that speeds up (register-log -> notation). Sidebar: the SN76489 across ColecoVision/Sega/BBC. ## Ch. 21 — Keyboard, Joysticks, and the TMS9901 (drafted session 7; ~4,000 words + code ≈ 16 pp; status: draft complete; inplib machine-verified on BENCH99 at commit bb565e4; code in code/ch21/)

## Ch. 23 — Console ROM Services: GPLLNK, XMLLNK, and Floating Point (drafted session 7; ~3,600 words + code ≈ 16 pp; status: draft complete; the FP-vs-fixed-point anchor (FXMUL) machine-verified on BENCH99 at commit bb565e4; code in code/ch23/)
- The console ROM/GROM is a LIBRARY you can borrow (char-set loaders, floating point, cassette, KSCAN) through two doorways: **GPLLNK** (BLWP linkage -> a GPL-side routine in GROM, via the interpreter — char sets, cassette) and **XMLLNK** (-> a ROM machine-code routine by table code or direct address — faster, the FP door). Pattern = linkage + a DATA word naming the routine + scratchpad setup. Bridge: calling firmware = cooperating with a program that shares your memory (what a "system call" really is).
- GPLLNK contract: it runs the GPL interpreter (workspace >83E0, status >837C) — set up the exact scratchpad, don't trample GPL state, return the machine as lent. The project GROM really implements these (e.g. the >004A char-set loader). XMLLNK stays in machine code (faster, no interpreter tax).
- **Floating point** = radix-100: 1 exponent byte + 7 base-100 mantissa bytes = 8 bytes (decimal-friendly, typed==stored). Operates on FAC (>834A) + ARG (>835C): +/-/x/div/compare + CIF/CFI + number<->string. TI BASIC = this + a parser.
- **Judgment** (the book's counsel): FP is general but hundreds-to-1000+ cyc/op — a luxury in a 50k frame. Games use fixed point: FXMUL (8.8, one MPY+shift) VERIFIED (1.5*2=>0300, 2.5*2.5=>0640, 0.5*0.5=>0040) at ~220 cyc — several times cheaper. FP only for real range/decimal-precision needs (why Part III used 8.8 throughout).
- **Environment honesty** (§23.6, the key caveat): GPLLNK/XMLLNK are E/A UTILITIES — E/A installs the vectors. A bare cartridge has none -> `BLWP @GPLLNK` crashes. Fix: re-implement SHIMS (Ch. 35) or avoid the services (ship own font + fixed-point — what lib99 does). Lab: fxcalc (fixed-point, verified) + an FP calculator (E/A-hosted, benchmarked against FXMUL). Field Notes: radix-100 spotted in VDP RAM. Seeds: GPL (Ch. 26), TI BASIC (Ch. 28), cart shims (Ch. 35), optimization (Ch. 37).

## Ch. 25 — GROM: The Strangest Memory You'll Ever Address (drafted session 7; ~4,300 words + code ≈ 14 pp; status: draft complete; gromlib + GROMDUMP machine-verified on BENCH99 at commit c970196; code in code/ch25/)
- A **GROM** (TI's TMC0430) is a 6 KiB ROM with NO address bus — an internal 16-bit counter you seek + a stream you read, "like cueing a tape and pressing play." Vignette = a ROM you cannot point at. Bridge: serial memory = a file (seek + sequential read); TI used it for the OS + flagship software, for density/cost/proprietary control (the 1979 price war). Everything in Part VI lives in GROMs.

Read = aim (address hi then lo to >9C02) then stream from >9800. **The prefetch**: aiming at A immediately buffers mem[A] + advances the counter to A+1 — first read returns mem[A] (right) but reading the address back shows A+1 (the pipeline, not a bug). Verified: gromlib reads the console header AA 02 00 00 00 00 00 00 13 10 13 20 (= the ROM file) from ONE aim.

Console GROMs 0-2 = OS/menu/TI-BASIC; cart GROMs 3-7 scanned at boot. **16 parallel GROM BASES** (>9800/>9804/...) for peripherals (p-code card). **GRAM** = writable GROM-space (GRAM Kracker / FinalGROM 99 / the emulator) — the door to run our own GPL (Ch. 27); the idle write-data port is why we can author, not just read.

Lab: GROMDUMP checksums 256 GROM bytes to >50E7 (= the file's sum) — a GROM fingerprint tool.
