## Eight Kilobytes and a Window

A TI-99/4A cartridge is, electrically and in the memory map, almost absurdly simple: **8 kilobytes of ROM** that appear in the CPU's address space at **`>6000`–`>7FFF`**, a single 8 KiB window (Chapter 5). Plug in the cartridge, and those 8 KB are there; that is the whole of it. And 8 KB is *tiny* — a single screen of bitmap graphics is 6 KB (Chapter 15), a decent sound-and-music driver is a kilobyte, a font is another — so the obvious question is the one that makes cartridge engineering an art: **how did a real game, tens of kilobytes of code and data, ever fit in an 8 KB window?**

The answer is **bank-switching**, and it is the elegant, slightly mad trick at the heart of this chapter: you put *many* 8 KB banks of ROM on the cartridge, but only one is visible in the `>6000` window at a time, and you **switch** which one — by *writing* to the window, of all things — fast enough that the program flows across banks as if it were one large ROM. A 32 KB game is four banks swapped through one window; a modern megacart is hundreds. The whole discipline is making that work without the seams showing: keeping the code that *does* the switching present no matter which bank is live, laying out banks so the swaps fall at safe moments, and threading control across the boundaries. Get it right and a cartridge holds a world; get it wrong and the program vanishes the instant it swaps the ground out from under its own feet.

This chapter is that engineering, verified. We will build a real bank-switched cartridge and *watch it switch banks* on the bench — the write to the window, the new bank appearing, the resident code surviving the swap — because bank-switching is not folklore here; it is a mechanism the emulator implements and we can prove. And it opens Part VIII, "Cartridge and Software Engineering," because the cartridge is the *package* the machine's best software shipped in, and knowing how to fill it — banks, headers, initialization, the build — is knowing how to ship.

## The Bridge: When Storage Outgrows the Address Space

Bank-switching is the first solution to a problem that has recurred at every scale of computing since: **you have more stuff than you have address space to see it in**. The 9900 can address 64 KB; a cartridge wants to hold 512 KB; the window is 8 KB — so you *page*, swapping views of the large store through the small window. You have already met the pattern twice in this book — GROM's slots (Chapter 25) and SAMS's 4 KB mapper (Chapter 34) — and it is *everywhere* in computing's history: DOS's EMS/XMS memory (bank-switched RAM through a window, for the exact same reason — the 8086 address limit), the MMU and virtual memory (pages of a huge address space mapped through physical frames), overlay linkers (swapping code sections through a fixed region), dynamic library loading, even a modern GPU streaming textures through a fixed memory budget. The idea is one idea: **a small window onto a large store, remapped on demand.**

So learning cartridge bank-switching is learning the *ur-form* of paging, on the smallest, most visible instance: one 8 KB window, a handful of banks, a switch you can watch happen. The disciplines it forces — keep the "which page is mapped" machinery resident (never page out the pager), lay data so accesses cluster within a page, cross boundaries deliberately — are the *same* disciplines that govern virtual memory, overlays, and streaming today (a page fault is expensive; a cache miss is expensive; a bank switch mid-loop is a bug). When you engineer a bank-switched cartridge, you are doing, in miniature and in full view, what an operating system's memory manager does behind the curtain. The window is 8 KB and the year is 1982, but the problem — and the shape of the answer — is eternal.

## 35.1 What a Cartridge Is

Electrically, a cartridge is a ROM chip (or several) on a small board with an edge connector that mates with the console's cartridge port, wiring the ROM's data lines onto the CPU bus so it answers at `>6000`–`>7FFF`. From a programmer's eye, that is the entire schematic: **ROM in the `>6000` window**, plus — for bank-switched and hybrid carts — a little logic that decides *which* ROM bank the window shows, and possibly GROM chips too (Chapter 27's GPL cartridges). The console, at boot, looks for a cartridge exactly as it looks for a GPL program: it checks the window for the **`>AA` header**.

That header is the **ROM sibling** of Chapter 27's GPL cartridge header — the same `>AA` signature, the same idea of a program list — now for *machine-code* programs. At `>6000` a ROM cartridge places `>AA`, a version, and pointers to a **program list** (the menu entries, each with a name and an entry address) and a **power-up list** (code to run at boot, Chapter 28). So a machine-code cartridge introduces itself to the master menu identically to a GPL one — the console's scan (Chapter 28) finds the `>AA`, walks the program list, shows "`n FOR NAME`," and launches the chosen entry (a machine-code cartridge via `XML >F0`, the trampoline Chapter 29 verified). The header is the universal handshake once more: cartridge, DSR, GPL program, ROM program — all say `>AA` and a linked list, and the console discovers them all the same way. `libre99asm` in cartridge mode writes this header for you (auto-generating the `>AA` and pointing it at your `START`), which is why every cartridge this book builds is menu-discoverable by construction.

## 35.2 The 8K Problem and Bank-Switching Schemes

Eight kilobytes is not enough, so cartridges carry more ROM than the window shows and **switch banks**. The mechanism is delightfully counterintuitive: you select a bank by **writing to the ROM window** — a location that is read-only for *data* but whose *writes* are intercepted by the cartridge's bank logic to change which bank is mapped. The TI standard scheme, which the project implements and we verify, derives the bank from the write *address*: writing to `>6000 + 2N` selects bank `N` (precisely, the core computes `bank = (address >> 1) & (banks − 1)`, so the bank count is a power of two and the address's bit pattern picks the bank). A write of *any* value works — it is the address that matters, not the data — so `MOV R3,*R3` with `R3 = >6002` selects bank 1, and `R3 = >6000` selects bank 0. We prove it on the bench: a two-bank cartridge whose bank 0 and bank 1 carry different marker bytes, run, and the marker read back changes as the program writes the window:

```text
bench: load bankcart.bin (2 banks); run
  >8320  B0 B0  B1 B1  B0 B0
         bank0  ->w>6002->bank1  ->w>6000->bank0
```

The marker is `>B0B0` when bank 0 is mapped, `>B1B1` after a write to `>6002` maps bank 1, and `>B0B0` again after a write to `>6000` — bank-switching, verified through actual CPU writes.

The **catalog of schemes** builds from this. The simple **2-bank** style (a single bank bit) doubles a cartridge to 16 KB. The modern **378/379 "inverted" multi-bank** standard generalizes it to many banks (32 KB, then 512 KB, then megabytes) with more address bits selecting the bank — the scheme modern homebrew megacarts use, and the one the FinalGROM 99 (Chapter 34) speaks. **ROM+GROM hybrids** carry both ROM banks and GROM chips (Chapter 29's hybrid cartridges — GPL shell in GROM, 9900 core in banked ROM). And **RAM-bearing cartridges** (the Mini Memory lineage) put *writable* memory in the window — battery-backed RAM that survives power-off, so the cartridge is also storage. The project models the core scheme (the TI write-to-address bank select, verified) that all of these elaborate; the exotic multi-bank and RAM variants are the same idea with more bits or a writable chip, and where a specific board's scheme differs, capability and the board's documentation govern (R-12). The through-line: more ROM than the window, selected by a write, all resting on the mechanism we just watched work.

## 35.3 Bank-Safe Program Design

Bank-switching has one deep hazard, and the entire discipline of bank-safe design is built around it: **the code that switches banks lives in a bank**. If your program is executing in bank 0 and switches to bank 1, the instructions *under the program counter* just changed — the next instruction fetched is whatever happens to be at that address in bank 1, almost certainly not the continuation you meant. Switch banks carelessly and the program leaps into garbage. The solutions are a small, sharp toolkit.

The **resident common segment** is the foundation: a region of code that is **identical in every bank at the same addresses**, so that no matter which bank is mapped, that code is present and continues correctly. You place your bank-switching routine, your interrupt handler, and your critical dispatch in the resident segment (assembled into every bank), and a switch executed *from* resident code is safe because the resident code is still there afterward. The **trampoline** is the resident helper that performs a switch-and-call: it maps the target bank, calls into it, and — because the trampoline itself is resident — survives to map the caller's bank back on return. Our verified `bankcart` demonstrates the cleanest form: it copies the switching probe into **scratchpad RAM** and runs it *there*, so the switch happens from RAM, entirely outside the `>6000` window, and no bank change can disturb it — the `B @>8360` into RAM before any switch is the whole trick. **Per-bank vector tables** let each bank expose its routines at known resident addresses; **data banking** puts read-only data (music, levels, graphics) in banks the resident code pages in to copy from (never *executing* from a bank it is about to switch away). And the **bank-aware build map** — knowing which bank every symbol lives in, and keeping cross-bank references trampolined — is the bookkeeping that keeps it all straight (`libre99asm`'s per-bank assembly, §35.5). The rule, stated once: **never switch the bank you are running in without a resident foothold**, and the trampoline (in a resident segment or in RAM) is that foothold.

## 35.4 Startup Without the E/A Comfort Blanket

A cartridge boots into a **cold machine**. Unlike an Editor/Assembler program (Chapter 6), which runs in an environment E/A has already set up — a workspace, utility vectors, an initialized VDP — a cartridge is launched by the master menu (Chapter 28) into a console that has done only its own boot, so the cartridge must **initialize everything itself**. It must establish its **workspace** (`LWPI` to a scratchpad address — Chapter 24's fast island, since a game's workspace must be fast), claim its **scratchpad** regions (declaring which pad cells are its variables, clear of the console's live areas — Chapter 24's atlas), and bring the **VDP up from cold** (set the mode, load the pattern table, clear the screen — Part III's setup, not assumed done). And it cannot lean on E/A's utilities — `VMBW`, `DSRLNK`, the console services — as freely, because those live in the E/A environment a bare cartridge does not have (Chapter 23's honesty): a cartridge either brings its own (as `lib99` does — self-contained `vdplib`, `filelib`), or uses the console-ROM services carefully with the shims Chapter 23 developed. This is the natural habitat of those shims: a shipped cartridge that must run on a bare console (or one with only the console ROM) provides its own foundation, because nothing sets it up for you. The discipline is Chapter 24's and Chapter 23's, applied at ship time: know exactly what state the machine is in when your entry runs (cold, menu-launched), and build your world from there — workspace, pad, VDP, services — assuming nothing.

## 35.5 Build and Ship

A finished cartridge must become the several **files** the ecosystem loads, and the ideal is "one build, several targets." `libre99asm` builds the core artifact — an **8 KB bank** as a `.bin` (padded ROM image) or a `.ctg` (the project emulator's format, Chapter 5), with the `>AA` header auto-generated. For a **bank-switched** cartridge, the honest state (R-12): `libre99asm` assembles **one 8 KB bank at a time**, so a multi-bank cartridge is built by assembling each bank (with the resident segment common to all) and **concatenating** them into a multi-bank image — which is exactly how this chapter's two-bank `bankcart` was built (two 8 KB banks, `cat` into one 16 KB image) and how the bench loads it (auto-detecting the bank count from the file size, `banks = length / 8K`). A single-image multi-bank `.ctg`/`.rpk` **packager** — RLE-compressing and framing the banks per the `.ctg` format — is a clean roadmap item the book surfaces; until then, the raw concatenated image is the multi-bank build, and it runs. The **targets** are: `.ctg` for the project emulator, padded `.bin`s for the **FinalGROM 99** (Chapter 34's flash cart — copy to SD, run on real hardware), `.rpk` for **MAME**, and configurations for **Classic99**. **Version and reset etiquette** matters for a shipped cart — a reset should return to a clean state, and a version byte in the header helps tools — and **testing across console revisions** (Chapter 28's version differences) is the final gate: a cartridge that assumes one console's exact behavior may stumble on another, so test on the emulator, a real 4A, and the shelf emulators. Build once, package for each, test everywhere.

## 35.6 Multi-Program Cartridges and Menus

Because the header's program list is a *list*, one cartridge can offer **several programs** — multiple entries, each with a name and an entry address, all shown on the master menu ("`1 FOR GAME`," "`2 FOR EDITOR`," Chapter 27's mechanism). A multi-program cartridge is thus a small suite: a game and its level editor, a collection of tools, several games on one megacart — each a program-list entry the console discovers and launches. And a cartridge can present **its own menu** instead of (or above) the console's: its entry point draws a custom selection screen and dispatches to sub-programs itself, giving a branded front-end (the way a commercial multi-game cartridge shows *its* menu, not the console's bare list). Building one is the header work of §35.1 (multiple program-list entries) plus, for a custom front-end, a menu program (Chapter 14's `textlib40` UI) that `XML`s or branches to the chosen sub-program — and with bank-switching, each sub-program can live in its own bank, the resident menu paging in the selected one. The multi-program cartridge is where all of Part VIII's pieces meet: the header (discovery), bank-switching (capacity), the resident segment (the menu), and clean init (each program from cold).

## Lab 35 — DODGE as a Four-Bank Cartridge

The lab productionizes an earlier game into a real shipped cartridge, in `code/ch35/`.

**The bank-switch proof (`bankcart`)** is the chapter's verified core: the two-bank cartridge that demonstrates the switch (`>B0B0`/`>B1B1` markers, the write-to-`>6002` swap, the RAM-resident trampoline). Build and run it:

```sh
libre99asm code/ch35/bankcart.a99  --format bin -o build/bc0.bin
libre99asm code/ch35/bankcart1.a99 --format bin -o build/bc1.bin
cat build/bc0.bin build/bc1.bin > build/bankcart.bin   # a 2-bank image
```

Load it, run, and read `>8320`: `B0 B0 B1 B1 B0 B0` — bank-switching, on the bench.

**DODGE as a four-bank cart** is the productionization: take Chapter 16's `DODGE` and ship it as a **32 KB, four-bank cartridge** — the game kernel and `lib99` resident (in every bank, the code that always runs), the music and level data **banked** (paged in as needed), byte-identical behavior whether on the emulator or a flash cart. It is the whole chapter applied: a real game, laid out across banks with a resident common segment, its data banked, built as a concatenated four-bank image, running from the `>6000` window as if it were one large ROM. Building it teaches the layout discipline (what must be resident, what can bank), and running it — same behavior, emulator and hardware — is the proof that a cartridge holds far more than its 8 KB window shows.

> **Field Notes — A modern megacart's bank map.** With a homebrew author's permission, tour the bank map of a modern TI megacart — a cartridge holding *hundreds* of kilobytes across dozens of banks via the 379 scheme (§35.2). Its map is a document of the discipline: a small **resident segment** (the bank-switch trampolines, the interrupt handler, the core dispatch) repeated in every bank; a **menu bank** (the front-end that lists the cartridge's programs); and then bank after bank of **content** — whole games, each in its own bank or spanning several, paged in by the menu. Reading such a map — which symbols are resident, where each program lives, how the trampolines cross the boundaries — is reading cartridge engineering at its modern peak: the same 8 KB window this chapter started with, now a keyhole onto half a megabyte, managed by exactly the resident-segment-and-trampoline discipline §35.3 taught, scaled up by people who never stopped refining it. The megacart is the 8K problem, solved so thoroughly that the window's smallness becomes invisible — which was always the goal.

## Summary

A cartridge is **8 KB of ROM in the `>6000`–`>7FFF` window**, introducing itself to the master menu with the **`>AA` ROM header** (the machine-code sibling of Chapter 27's GPL header — same signature, same program/power-up lists, same discovery, launched via `XML >F0`). Since 8 KB is far too small for a real game, cartridges carry many banks and **bank-switch**: you select a bank by **writing to the window**, the TI scheme deriving the bank from the write *address* (`bank = (addr >> 1) & (banks − 1)`; the value is irrelevant) — machine-verified on the bench, a two-bank cart's marker reading `B0B0 → (write >6002) → B1B1 → (write >6000) → B0B0`. The scheme catalog runs from 2-bank to the **379 multi-bank** megacarts (hundreds of KB), with ROM+GROM hybrids and RAM-bearing carts as variants. The central discipline is **bank-safe design**: because the switching code lives in a bank, you keep a **resident common segment** (identical code in every bank), use **trampolines** (resident switch-and-call helpers) — cleanest of all, run the switch from **scratchpad RAM**, entirely outside the window (as the verified `bankcart` does) — and bank **data**, never executing from a bank you are about to swap. A cartridge starts from **cold** (menu-launched into an uninitialized console), so it establishes its own workspace, scratchpad, and VDP and cannot lean on E/A's utilities (Chapters 23–24's shims in their habitat). The **build** is "one bank at a time, concatenated" (R-12: `libre99asm` assembles single 8 KB banks; the multi-bank image is their concatenation, which the bench loads by size — a single-file `.ctg`/`.rpk` packager is a roadmap item), targeting `.ctg`, FinalGROM `.bin`, MAME `.rpk`, and Classic99. And because the header's program list is a list, a cartridge can hold **multiple programs** with its own menu. Bank-switching is the *ur-form of paging* — a small window onto a large store, remapped on demand — the same idea as GROM slots, SAMS, EMS, and virtual memory, learned here at its most visible: an 8 KB window you can watch a game flow through, verified switch by switch. Part VIII begins where the software ships: in the cartridge, engineered to make a world fit through a keyhole.
