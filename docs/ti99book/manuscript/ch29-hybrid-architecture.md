## Two Minds, One Program

We have spent Part VI on the machine's second mind — GPL, the interpreted bytecode in which TI wrote the operating system and most of its software — and Parts II through V on its first, the TMS9900 itself. The temptation is to see them as rivals: GPL *or* assembly, the slow-but-compact language *versus* the fast-but-verbose one, a choice you make per program. That is not how the machine's best software was built. The finest TI-99/4A programs — the first-party action games, the polished cartridges — were **hybrids**, deliberately: a **GPL shell** handling the menu, the title screen, the text, the data-driven flow, wrapped around a **9900 machine-code core** handling the parts that had to run fast — the motion, the collision detection, the sound service, the inner loop. Neither language alone could have made them. GPL alone would have been too slow for the action; assembly alone would have been too bulky and slow to *write* for the shell. Together, they were exactly right.

This chapter is the seam. It is where Part VI's GPL meets Parts II–V's assembly, and the meeting is not metaphorical — it is a specific mechanism, a GPL instruction called **`XML`** that reaches out of the interpreter and runs 9900 machine code, and its mirror, **`GPLLNK`** (Chapter 23), that reaches from 9900 code back into GPL. We watched the console itself use `XML` in Chapter 26 (`XML >1A`, live, at GROM `>0198`); we will now understand it as the doorway it is. And the chapter closes Part VI by building the thing the rest of the book will stand on: a **reusable hybrid skeleton** — a GPL shell plus a 9900 core, with the seam wired correctly — that is the chassis for the capstone projects of Part IX. Learn this pattern and you can build what TI built.

## The Bridge: The Two-Language Program You Already Write

You already build hybrids; you just do not call them that. A modern application is a high-level language — Python, JavaScript, Ruby — wrapped around native code for the hot paths: NumPy's C loops under Python's ergonomics, a game's C++ engine under a Lua scripting layer, a web app's WASM module under its JavaScript. The pattern is universal and deliberate: write the *bulk* of the program in the language that is pleasant and fast to develop (and fast *enough* for most of the code), and drop to a native language for the small, hot core where performance actually matters — crossing the boundary through a foreign-function interface. You choose the high-level language for the 95% that runs at human speed and the native language for the 5% that runs in the inner loop, and an FFI stitches them.

The TI hybrid is that architecture, in 1981. GPL is the "high-level" language — pleasant, compact, fast to write, fast enough for the shell — and 9900 assembly is the "native" core for the hot loop, with `XML` and `GPLLNK` as the foreign-function interface between them. Every instinct you have about when to drop from Python to C applies here unchanged: profile, find the hot loop, rewrite *that* in the fast language, keep the rest in the ergonomic one, and mind the boundary-crossing cost. Studying the TI hybrid is studying the FFI pattern at its origin, on a machine small enough that you can see both languages, the boundary, and the exact reason the boundary exists — the interpreter tax of Chapter 26, which is the same reason you leave Python for the hot loop today.

## 29.1 The House Pattern, and Its Economics

The TI house pattern is a division of labor by *speed requirement*. Everything that runs at human speed — the title screen, the menu, the between-level text, the score display, the attract mode, the data-driven game setup — is **GPL**, because GPL is compact and quick to write and the interpreter tax (Chapter 26) is invisible at human speed. Everything that runs at machine speed — the per-frame sprite motion, the collision checks, the sound-list service, the scroll, the inner game loop inside the 50,000-cycle frame (Chapter 17) — is **9900 machine code**, because there the interpreter tax is fatal and only native speed fits the frame. The shell is GPL; the core is assembly; the seam is `XML`.

The economics behind the pattern are specific to the machine's memory, and they cut two ways. **GPL code lives in GROM** — compact bytecode, and GROM was the cheap, dense memory (Chapter 25) — so a large shell (lots of text, menus, data) costs few GROM bytes and no scarce RAM. **9900 code lives in ROM** at `>6000` (the cartridge ROM of Chapter 35) — larger per operation, but fast — so the machine-code core costs more bytes per function but buys the speed the shell cannot. A cartridge designer thus had a real budget decision: GROM bytes for the compact, slow shell versus ROM bytes for the bulky, fast core, and the house pattern is the optimum — put as much as possible in cheap, compact GPL/GROM, and spend the expensive fast ROM only on the hot core. It is the identical calculus as "keep the codebase in Python, spend C only where profiling says," expressed in two kinds of 1981 memory chip: the shell is cheap and roomy, the core is dear and fast, and you buy each where it pays.

## 29.2 Calling Assembly from GPL: `XML`

The doorway from GPL to 9900 machine code is **`XML`** — eXecute Machine Language — a GPL opcode (Chapter 26) carrying a one-byte code that selects which machine-code routine to run. When the interpreter hits `XML nn`, it looks the code up and transfers control to the corresponding 9900 routine, which runs *natively* (at full CPU speed, no interpreter tax), does its work, and returns to the interpreter, which resumes the GPL stream. It is the FFI call: GPL says "run native routine `nn`," the native routine runs, control comes back.

We have seen the console itself do this, verified. `console.gpl`'s power-up scan calls `XML >19` ("find the next card with a power-up routine") and `XML >1A` ("call it") — reaching machine-code service routines from GPL — and Chapter 26 watched the interpreter execute `XML >1A` live at GROM `>0198`. Most consequentially, the master menu launches a **machine-code program** with `XML >F0`, the ML-launch trampoline (execution-verified, per `console.gpl`'s RECON notes): when you select an assembly cartridge from the menu, the GPL menu code does `XML >F0` to jump into the cartridge's 9900 entry — the hybrid's defining act, GPL dispatching to assembly. The `nn` codes come in two kinds: the **ROM routine table**, a fixed set of console-provided machine-code services (the `>19`/`>1A`/`>F0` kind) that any GPL program can call, and **user `XML` vector tables** — entries a program installs (in the scratchpad, at the console-defined vector addresses) so its *own* GPL can `XML` into its *own* 9900 core. That second kind is how a hybrid cartridge wires its shell to its core: the ROM core installs its routine addresses into the `XML` vector table at startup, and thereafter the GPL shell reaches the core with a plain `XML`. Parameters cross the boundary the way Chapter 23's links established — through the scratchpad and the GPL/CPU registers both sides agree on — because GPL's variables and the 9900's workspace both live in the same scratchpad, so a value the GPL shell writes to a pad word is a value the 9900 core reads from that same word. The boundary is real but thin: one instruction, a shared scratchpad, a native routine at full speed.

## 29.3 Calling GPL from Assembly: `GPLLNK`, Re-examined

The doorway swings both ways. Chapter 23 built `GPLLNK` from the 9900 side — the mechanism by which assembly code calls a GPL routine (a console service written in GPL) — and from the GPL side it is the mirror of `XML`: where `XML` lets GPL run native code and return, `GPLLNK` lets native code run GPL and return. A 9900 core that wants a service the console provides *in GPL* — a character-set load, a formatted screen operation, a floating-point routine (Chapter 23) — calls `GPLLNK` with the GPL entry it wants, the interpreter runs that GPL, and control returns to the 9900 code. So a hybrid program has *both* doorways available: its GPL shell reaches its 9900 core with `XML`, and its 9900 core reaches GPL services (or its own GPL) with `GPLLNK`.

The thing to state once, as law, is the **state contract** in both directions, because a hybrid that gets it wrong corrupts one language from the other. Crossing from GPL to 9900 via `XML`: the native routine must preserve what the interpreter needs — the GPL workspace (`>83E0`), the GPL program counter and data-stack pointer, the status byte (`>837C`) — or restore them before returning, exactly as a subroutine preserves its caller's registers (Chapter 9), because the interpreter resumes assuming its state survived the call. Crossing from 9900 to GPL via `GPLLNK`: the caller must set up what the GPL side expects (the pad-based parameters, the workspace) and accept that the GPL routine uses the interpreter's state, and both sides must respect the shared scratchpad — the GPL workspace and the ISR cells (Chapter 24) are off-limits to a core that wants to coexist with a live interpreter. The contract is the same principle Part V stated for every shared resource: what you borrow across the boundary — registers, workspace, pad, status — you return unchanged. Stated once here, it governs every hybrid: the seam is safe exactly when both sides honor the other's state.

## 29.4 Packaging Hybrids: ROM + GROM Cartridges

A hybrid cartridge carries *both* memories: **GROM** chips for the GPL shell (Chapter 25) and **ROM** at `>6000` for the 9900 core (Chapter 35), on one cartridge board, and packaging one is a matter of getting both headers and the initialization order right. The **GROM** carries the GPL program with its `>AA` header and program list (Chapter 27), so the master menu discovers and lists the cartridge; the **ROM** carries the 9900 core with *its* header (the ROM-cartridge header of Chapter 35, also `>AA`-based). The **initialization choreography** is the question of *who initializes whom*: typically the menu launches the GPL shell (the GROM program), and the shell, at startup, installs the ROM core's `XML` vectors (§29.2) so it can call the core — or, for an assembly-led cartridge, the menu launches the ROM core via `XML >F0` and the core calls back into GPL for services. Either way, one side is the entry the menu dispatches to, and that side brings the other online: the GPL shell installs the core's vectors, or the ROM core sets up its GPL calls. The build system assembles the 9900 core with `libre99asm` (to ROM at `>6000`) and the GPL shell with `libre99gpl` (to GROM), and packages both into one cartridge image — the "one build, several targets" pipeline whose cartridge-emission step is the §27.5 roadmap item. Until that step is automated, a hybrid is packaged with the shelf cartridge tools (Chapter 27), but the *architecture* — two memories, two headers, one initialization order — is exactly as described, and it is what every first-party hybrid cartridge did.

## 29.5 How a First-Party Game Divided Its Labor

Put the pattern against a real game — reconstructed behaviorally, in the debugger, the way this book reads software it does not have the source to. Take a canonical first-party action cartridge (a *Parsec*-class game: an action title with a polished shell). Boot it and watch, and the division of labor is legible in the traces. The **attract mode and title** — the animated logo, the scrolling instructions, the high-score display — are GPL: trace them and you see the interpreter's GROM fetches (Chapter 26's `gromlog`), the `MOVE` cascades painting the screen, the human-speed flow. Press start, and the game proper hands off to the **9900 core** — trace the inner loop and you see native 9900 instructions running per-frame: the ship's motion, the enemy sprites' movement and the law-of-four juggling (Chapter 16), the collision detection, the sound-list service (Chapter 19) feeding the PSG each frame — no interpreter in the hot path, because the frame budget forbids it. And at the seams — level transitions, the between-wave text, the game-over screen — control returns to **GPL** for the human-speed interludes, then back to the 9900 core for the next wave. The `XML` doorway is where the hand-offs happen: the GPL shell `XML`s into the core to start a wave; the core, when the wave ends, returns to the shell. Reconstructing this from the outside — GPL where the traces show interpreter fetches and human-speed flow, 9900 where they show native inner loops at frame rate — is a skill this book has built chapter by chapter, and here it reads a whole game's architecture: shell and core, seamed by `XML`, exactly the house pattern.

## 29.6 A Reusable Hybrid Skeleton

Part VI ends by building the chassis the rest of the book drives. The **hybrid skeleton** is a minimal, correct template: a **GPL shell** (a program header the menu discovers, a title screen, a menu, the human-speed flow) plus a **9900 core** (a workspace, an inner loop, the fast kernels), wired at the seam (the shell installs the core's `XML` vectors; the core preserves the interpreter's state; parameters pass through named pad words). It is `quizmaster.gpl` (Chapter 27's GPL shell) joined to a `lib99` 9900 core (Parts III–V's verified kernels — `spritelib`, `sndlib`, `inplib`) through `XML`, with the state contract of §29.3 honored on both sides. This skeleton is deliberately the **chassis for Part IX**: the capstone games and applications are built *on* it — a GPL shell for their menus and text, a 9900 core for their action, the seam already correct — so a reader reaching Part IX does not re-derive the architecture but instantiates it. Everything Part VI taught converges here: the GROM and its GPL (Chapters 25–26), the header and the toolchain (Chapter 27), the boot and menu that launch it (Chapter 28), and now the seam that joins the two languages into one program (Chapter 29). The skeleton is the deliverable — the thing you carry out of Part VI — and it is, in one artifact, how the TI-99/4A's best software was built.

## Lab 29 — Rebuild the Round in 9900

The lab makes the hybrid's payoff measurable, extending Chapter 27's QUIZMASTER.

**The build.** Keep QUIZMASTER's GPL shell — its title, its menu entry, its question display — *byte-identical* (it runs at human speed; GPL is right for it). Rewrite its one **timing-critical round** — say, a fast-reaction timed round where the display must update many times a second — as a **9900 core**, invoked from the GPL shell with `XML`. The shell installs the core's vector at startup and `XML`s into it when the timed round begins; the core runs the fast loop natively and returns to the shell when the round ends.

**The measurement.** Measure the round both ways — as GPL (Chapter 26's interpreter tax, traced with `gromlog` and `s`) and as the 9900 core (Chapter 22's `cycles`, the native inner loop) — and the difference is the chapter's whole argument in numbers: the same round, roughly an order of magnitude faster in the core, while the shell around it is unchanged. You will have done, in miniature, exactly what a first-party team did: profiled a program, found the hot round, dropped *it* to assembly through `XML`, and left the rest in GPL — and measured the win. That measured contrast, GPL shell cost versus 9900 core cost on the same task, is the hybrid architecture justified on the bench, and the template you carry to Part IX.

> **Sidebar — Extended BASIC is itself a hybrid.** It is worth one sentence of respect: TI Extended BASIC — the enhanced BASIC many TI users actually programmed in — is itself a hybrid, a GPL/assembly extension of the TI BASIC of Chapter 28 that added assembly-accelerated routines and, crucially, `CALL LINK`, a doorway from BASIC to the user's own 9900 machine-code subprograms (the one fenced `CALL LINK` page this book allows appears in Chapter 36). That the machine's most popular programming environment was *also* built on the shell-plus-core pattern — a high-level language reaching into assembly for speed — is the final proof that the hybrid was not a trick a few first-party teams used but *the* architecture of the platform, from the operating system down to the language the users typed in. And with that, we move on.

## Summary

The TI-99/4A's best software was **hybrid** — a **GPL shell** (menus, title, text, human-speed flow) wrapped around a **9900 machine-code core** (motion, collision, sound, the inner loop) — because neither language alone sufficed: GPL is too slow for the action (the Chapter 26 interpreter tax), assembly too bulky and slow to write for the shell. The division is by **speed requirement** (human-speed → GPL; machine-speed → 9900) and the **economics** back it: GPL code is compact and lives in cheap, dense **GROM**, 9900 code is fast and lives in dearer **ROM** at `>6000`, so a designer spends cheap GROM on the roomy shell and expensive ROM only on the hot core — the same "Python for the bulk, C for the hot loop" calculus, in two kinds of 1981 memory. The seam is **`XML`** — a GPL opcode that runs a native 9900 routine and returns (the FFI call), selecting the routine by a one-byte code from either the **ROM routine table** (console services like `XML >19`/`>1A`, and `XML >F0`, the verified machine-code-launch trampoline the menu uses) or a program's own **`XML` vector table** (installed in the scratchpad, wiring a cartridge's GPL shell to its 9900 core). Its mirror is **`GPLLNK`** (Chapter 23), which lets 9900 code run GPL and return, so a hybrid has both doorways. The **state contract** governs both: what you borrow across the boundary — the GPL workspace `>83E0`, the program counter and data-stack pointer, the status byte `>837C`, the shared scratchpad — you return unchanged, or you corrupt one language from the other. A hybrid **cartridge** carries both memories (GROM + ROM), each with its `>AA` header, and an initialization order in which the menu-dispatched side brings the other online. First-party games divide their labor exactly so — GPL attract/title/interludes, 9900 inner loop, seamed by `XML` — legibly reconstructable in the debugger. And Part VI ends by building the **reusable hybrid skeleton** (Chapter 27's GPL shell + a `lib99` 9900 core, wired through `XML`, the contract honored) — the chassis for Part IX — proving on the bench that the same round is roughly an order of magnitude faster in the core while the shell is unchanged. Even Extended BASIC is a hybrid (`CALL LINK`), the shell-plus-core pattern reaching from the operating system down to the language users typed in: not a trick, but *the* architecture of the machine.
