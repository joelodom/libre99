## A World in a Whisper of Memory

The numbers deserve a moment of vertigo. A complete, polished TI-99/4A game — a scrolling shooter with music and speech, an RPG with monsters and maps and dialogue — ran in tens of kilobytes: a 32 KB cartridge, 256 bytes of scratchpad, 16 KB of VRAM, maybe 32 KB of expansion RAM if you were lucky. The whole thing. A single uncompressed screenshot of a modern game is larger than every byte a TI game ever occupied. And yet these games were *complete* — they had title screens and difficulty curves and high-score tables and attract modes, they were fun, they shipped — which tells you something important: the constraint was not a wall that stopped good software, it was a **discipline that shaped it**. When you have 32 KB, you cannot be sloppy; every byte is budgeted, every structure is chosen, every feature is weighed. The result, done well, is architecture of a clarity that abundance rarely produces.

This chapter is that discipline, and it is the pivot of the whole book — the place where the parts (CPU, VDP, sound, input, storage, cartridge) become a *program*. It teaches how to **budget** memory as a first-class design act, how to use the machine's odd memories (VRAM as a warehouse, disk as overlay storage) to hold more than fits, how to structure a game's objects and states so they are fast and clear, how fixed-point and data-driven design let a small program do large things, and how to save state across sessions. And it delivers **SKELETON99** — the official project template, a verified skeleton with a phase state machine, an entity table, and fixed-point motion, that every capstone in Part IX instantiates. By the end you will not just know the pieces; you will know how to assemble them into a shippable program that fits, and you will have the chassis to do it on.

## The Bridge: Constraints and Clean Architecture

The idea that constraints improve design is not nostalgia; it is a working principle of modern engineering. **Embedded systems** programmers still budget every byte of a microcontroller's few kilobytes of RAM. **Game engine** architects obsess over memory layout and cache behavior because a modern CPU, for all its gigabytes, is starved for *fast* memory — so the game's hot data must fit in cache exactly as the TI's hot data must fit in the scratchpad. The **Entity-Component-System** pattern that dominates modern game architecture is, at its core, the TI's fixed-slot object table generalized: dense arrays of homogeneous data, iterated tightly, chosen for mechanical sympathy with the hardware. And the discipline of **data-driven design** — separating engine from content so designers can work without programmers — is exactly what *Tunnels of Doom* did in 1982 (Chapter 41) and what every modern engine does now.

So the TI's 32 KB is not a quaint limitation you are slumming in; it is a **teacher of the disciplines that still matter**, with the volume turned up so loud you cannot ignore them. When memory is abundant you *can* be sloppy and often are; when it is 32 KB you *must* budget, must choose dense data structures, must separate content from code — and those musts are the same practices that separate good modern software from bad, just enforced by the hardware instead of by discipline. Learning architecture here, under real scarcity, builds instincts that transfer directly upward: the programmer who can build a complete game in 32 KB understands memory, data structures, and system design in their bones, because the machine gave no other option. The constraint is the curriculum.

## 36.1 Memory Budgeting as Design

The first act of architecting a TI program is not writing code — it is filling in a **budget worksheet**, because on this machine memory is allocated before it is used, deliberately, or it runs out mid-development in a way that forces a painful redesign. The worksheet accounts for every kind of memory the machine has, each with its own size and character: **CPU scratchpad** (256 bytes, fast — the workspace and hottest variables, Chapter 24), **CPU expansion RAM** (up to 32 KB, medium speed — the main program and data, Chapter 5), **VRAM** (16 KB, port-accessed — the screen tables plus a warehouse, §36.2), **GROM** (for GPL programs, Chapter 25), **cartridge banks** (8 KB each, ROM — code and read-only data, Chapter 35), and **disk** (overlay and save storage, §36.3). You fill it in per program, and the shape of the fill *is* the architecture.

Three archetypes show the range. A **cartridge action game** budgets: kernel and `lib99` resident in a base bank, music and level data in other banks (Chapter 35), the entity table and hot variables in the scratchpad, the screen and a sprite-pattern warehouse in VRAM, no disk. A **disk RPG** budgets: a resident engine in expansion RAM, quest data streamed from disk as overlays (§36.3), the party and world state in expansion RAM, dialogue text paged through VRAM, save games to disk. A **productivity tool** budgets: the editor resident in expansion RAM, the document in a VRAM-and-expansion buffer, files to disk, minimal VRAM beyond the 40-column screen. Each worksheet is different, and each is *decided up front* — how big can the level be? how many entities at once? does the text fit or must it be paged? — because those answers determine the data structures, and the data structures determine the code. Budgeting is not accounting done after the fact; it is the design, done first, in the currency the machine actually charges: bytes.

## 36.2 VRAM as a Data Warehouse

The VDP's 16 KB of VRAM holds the screen — but the screen tables (Part III) rarely use all of it, and the leftover is a **warehouse**. In Graphics I, the name table, pattern table, color table, and sprite tables together occupy a few kilobytes; the rest of the 16 KB is free VRAM you can park *anything* in, reached through the VDP ports (Chapter 12). Programs use it heavily: parking **tables** (lookup tables, level maps, tuning data) that the CPU pages into scratchpad or expansion as needed; parking **graphics** not currently displayed (extra sprite patterns, alternate character sets swapped in per level); even parking **rarely-run code** that is copied to CPU RAM to execute when needed. VRAM is, in effect, a second memory bank — 16 KB of storage the CPU cannot execute from or address directly, but can read and write a byte at a time through the ports.

The trade-off is **access cost**, and it decides when the warehouse is brilliant and when it is a trap. VRAM is *slow* to touch — every access is a port protocol (set the address, read/write bytes, Chapter 12), far more expensive than a CPU RAM access — so it is **brilliant** for cold data (things touched rarely: a level loaded once per level, a lookup table copied to fast RAM once) and a **trap** for hot data (things touched per-frame: never put an entity's per-frame variables in VRAM, or the port overhead will swamp the frame budget). The rule is the memory hierarchy the whole machine teaches: **hot data in the scratchpad, warm data in expansion RAM, cold data in the VRAM warehouse (or disk)** — and knowing which is which is the skill. Use VRAM as bulk cold storage (its strength) and never as working memory (its trap), and 16 KB of otherwise-idle space becomes a genuine asset, holding the maps and tables and spare graphics a 32 KB program could not otherwise afford.

## 36.3 Overlays from Disk

When a program is larger than memory — a disk RPG whose engine, all its monsters, all its maps, and all its text will not fit in 32 KB at once — it uses **overlays**: it keeps a **resident** core in memory always and **loads** the rest from disk in phases, swapping sections in and out as the program moves through its stages. The architecture is **phase-based**: the program has distinct phases (title → dungeon → combat → shop → …), and each phase's code and data are loaded from disk when the phase begins, over the memory the previous phase used — so at any moment only the current phase plus the resident core occupies memory, and the total program can be many times larger than RAM. The **loader** (built on Chapter 31's `filelib` — LOAD a PROGRAM image, or READ data records) brings each phase in; the **resident core** — the parts every phase needs — stays put: `lib99` (the library every phase calls), the loader itself (which must survive to load the next phase — the same "never page out the pager" rule as Chapter 35's resident segment), the game-state machine (§36.4), and the persistent world state.

The discipline is deciding **what is resident and what overlays**, and it is the disk analogue of Chapter 35's bank layout: resident = small, universal, always-needed (the library, the loader, the state); overlaid = large, phase-specific, loaded on demand (a dungeon's map, a combat resolver, a shop's inventory). Keeping `lib99` resident is the key move — every phase calls `vdplib`, `spritelib`, `filelib`, so the library lives in the resident core and each overlay is just the phase's unique code and data, small because the shared machinery is already there. Overlays are how the RPG of Chapter 41 fits a world far larger than 32 KB into a machine with 32 KB: not all at once, but phase by phase, resident core plus current overlay, the disk as the backing store for a program too big to hold.

## 36.4 Entity and State Architecture

A game's *objects* — the player, the enemies, the bullets, the items — and its *modes* — title, play, game-over — need structure, and the TI's answer is two patterns SKELETON99 makes concrete and verified. The **fixed-slot object table** is the entity structure: a fixed-size array of fixed-size slots, each slot an entity's data (position, velocity, type, a live flag), iterated as a tight loop. SKELETON99's is eight slots of eight bytes (`x.hi x.lo y.hi y.lo dx dy type live`), and its `STEP` routine is the canonical entity loop — walk the slots, skip the dead ones, advance the live ones — verified on the bench (an entity seeded at position 1.00 with velocity 0.25 advances to 1.25 in one step). It is deliberately *not* a linked list of heap objects (the modern instinct); it is a **dense array**, because on this machine (as on a modern CPU's cache) a dense array iterated linearly is dramatically faster than pointer-chasing scattered allocations — the fixed-slot table is the Entity-Component-System pattern's ancestor, chosen for the same mechanical-sympathy reason. Fixed slots also mean fixed memory (no allocation, no fragmentation — you budget N entities and that is that) and simple lifetime (a live flag, not a free list).

The **game-state machine** is the mode structure: a `STATE` variable naming the current phase, and a **dispatch table** mapping each state to its handler, so the main loop is "look up the current phase's routine and run it." SKELETON99's is verified: `STATE = 0` dispatches to the title phase (signature `>7171`), `STATE = 1` to the play phase (signature `>5050`), through a jump table (`PTAB`), exactly as a real game's loop dispatches title/play/over each frame. This is the clean way to structure a program with modes — not a tangle of flags and conditionals, but an explicit state and a table of handlers, each phase a self-contained routine, transitions a matter of changing `STATE`. Together the two patterns are the spine of every game: a fixed-slot entity table holding the objects, a state machine sequencing the modes, both fast, both dense, both fixed in memory — and both in SKELETON99, proven, ready for Part IX to flesh out.

## 36.5 Fixed-Point Doctrine

The 9900 has no floating point (Chapter 8), but games need fractional positions and velocities — an object moving 1.5 pixels per frame — so the house arithmetic is **fixed-point**, and the standard is **8.8**: a 16-bit value split into an 8-bit integer part and an 8-bit fraction (256ths), so `>0140` is 1.25 (Chapter 23's `fxcalc`, and SKELETON99's verified motion). 8.8 is the default because it fits a register, covers the screen's coordinate range with useful sub-pixel precision, and its arithmetic is cheap — adding a velocity is a plain `A` (as `STEP` does), and the integer part (the pixel to draw at) is just the high byte. Other splits win in other places: **12.4** (12 integer bits, 4 fraction) when you need more range and less precision (large world coordinates), **4.12** when you need more precision and less range (fine increments, trig results) — the split is a design choice per quantity, but 8.8 is the workhorse.

**Angles and trig** get the same treatment. Rather than degrees or radians, games use a **256-unit circle** (a "byte angle": 256 = full turn), because a byte angle wraps naturally (angle + turn rate, mod 256, is a free byte add) and indexes a **sine table** directly. The sine table — 256 entries of the sine of each byte-angle, in 8.8 or a signed fraction — is precomputed (generated once, by a tool or an init routine) and stored, so `sin(angle)` is a table lookup, not a computation: to move an entity at a byte-angle, look up sin and cos (cos is sin shifted by 64), scale by speed (a fixed-point multiply, `fxcalc`'s `FXMUL`), and add to the position. This is **table-ism** (Chapter 37): precompute the expensive trig into a table, and the runtime is a lookup and a multiply. Fixed-point plus byte-angles plus sine tables is the entire math toolkit for motion — no floats, no runtime trig, just adds, shifts, table lookups, and the occasional multiply — and it is enough for any game the machine ever ran, verified in miniature by SKELETON99's 8.8 `STEP`.

## 36.6 Data-Driven Everything

The most important architectural decision a TI game makes is to be **data-driven**: to separate the *engine* (the code that plays) from the *content* (the levels, the enemies, the tuning, the text), storing the content as **data** the engine interprets rather than as code. A level is not a hand-coded routine that draws it; it is a **level format** — a compact binary description (tile map, enemy placements, exits) the engine reads and renders. An enemy's behavior is not a bespoke function; it is a **tuning table** entry (speed, health, a behavior selector) the engine's generic logic reads. The game's text is not string literals scattered through code; it is a **text bank** the engine indexes. This separation is transformative at 32 KB: the engine is written once and small, and the content — which is most of a game — is dense data, far more compact than equivalent code, and *changeable without reprogramming* (a new level is new data, not new code). It is also what lets the Chapter 38 asset pipeline exist: if content is data, tools can *generate* it (a map editor emits the level format, a script emits the tuning table), so designers work in data while programmers work in the engine.

SKELETON99 hooks this in: its phase dispatch is *itself* data-driven (the `PTAB` table drives the state machine — verified), and its entity `STEP` reads each entity's data (velocity, live flag) rather than special-casing entities — the generic-engine-over-data pattern in miniature. A full game extends it: the wave director reads enemy scripts (Chapter 39), the RPG plays a quest file (Chapter 41), the tuning lives in tables the designer balances (Chapter 36's worksheets become spreadsheets). Data-driven design is the single highest-leverage architecture choice on a memory-starved machine — it makes the code small and the content compact and changeable — and it is the thread from here through every capstone: **write an engine that plays databases**, and the databases are where the game lives.

## 36.7 Saving State

A game that cannot remember is diminished — no high scores, no saved progress, no configuration — so **persistence** is architecture too, with three tiers by what storage is present. With a **disk** (Chapter 31), state saves as files: high scores and configuration as small DV80 or INTERNAL files, saved games as PROGRAM-image or record files (Chapter 41's versioned save slots), written with `filelib` and read back at startup. With only a **cassette** (Chapter 33), state saves as a tape block — slower and clumsier, but the same idea (a data block out, a data block in). And with **no storage at all** — a bare cartridge on an unexpanded console — the fallback is the era's clever trick: **codes and passwords**. The game encodes the player's progress (level reached, items held, score) into a short alphanumeric **code** it displays at a checkpoint; the player writes it down, and next session types it in, and the game **decodes** it to restore the state — persistence with no storage, the state carried in the player's notebook. The codes are *computed and verified* (a checksum or a light cipher so invalid codes are rejected and the encoding is not trivially guessable), which is a small, satisfying piece of engineering: pack a game's progress into a dozen characters, robustly. The tier you build depends on the target (a disk RPG saves to disk; a bare-cartridge arcade game uses codes), but every finished game persists *something* — a high score at least — because a game that forgets everything the moment it is off is a demo, not a game. Persistence, in whatever tier the hardware allows, is part of shipping.

> **Sidebar (fenced) — The Extended BASIC `CALL LINK` ABI, in one page.** This book does not teach Extended BASIC — but half the commercial TI catalog shipped as XB programs with an assembly-language core reached through **`CALL LINK`**, so the assembly side of that bridge earns exactly one page (and no more). `CALL LINK("NAME", arg1, arg2, …)` in an XB program calls a machine-code subprogram you loaded (via XB's assembly support), passing the arguments; on the assembly side, your subprogram finds them through the XB **argument-passing ABI**: XB's `NUMREF`/`STRREF` utilities (whose addresses XB provides) let your code read a numeric or string argument by number, and `NUMASG`/`STRASG` let it write a result back into an XB variable — so a `CALL LINK` subprogram reads its inputs with `NUMREF`, does its fast work in 9900 code, and writes its outputs with `NUMASG`, and XB continues with the results. The subprogram is entered with a workspace XB sets up, must preserve XB's environment (the same borrow-and-return discipline as every hybrid, Chapter 29), and returns with `RT`. That is the whole ABI from the assembly side: read args by number, compute, assign results, return — the identical shell-and-core hybrid pattern of Chapter 29, with XB as the shell instead of GPL. It is why an XB game could have a fast assembly kernel: the same architecture, one language up. And that is the entire allowance — the ABI, one page, no XB taught; if you need the XB side, its manual awaits.

## Summary

A complete TI game fits in tens of kilobytes, and the **constraint is a discipline** that produces clean architecture. It begins with **memory budgeting as design** — a worksheet accounting for scratchpad (fast/tiny), expansion RAM (main), VRAM (16 KB, port-accessed), cartridge banks, and disk, filled in per program (cart action game, disk RPG, productivity tool), the shape of the fill *being* the architecture. The odd memories earn their keep: **VRAM is a warehouse** for *cold* data (maps, tables, spare graphics — brilliant) but never *hot* per-frame data (the port cost is a trap), following the hierarchy *hot in scratchpad, warm in expansion, cold in VRAM/disk*; and **disk overlays** let a program exceed RAM by keeping a resident core (`lib99`, the loader, the state machine) always present and loading phase-specific code/data on demand. The program's spine is two verified patterns from **`SKELETON99`**: a **fixed-slot entity table** (a dense array of fixed slots, iterated tightly — the ECS ancestor, chosen for speed and fixed memory; `STEP` advances live entities, verified) and a **game-state machine** (a `STATE` variable and a jump table dispatching title/play/over — verified `>7171`/`>5050`). Math is **fixed-point**: **8.8** as the house standard (`>0140` = 1.25, verified motion), 12.4/4.12 for other range/precision trades, and **256-unit byte-angles** indexing precomputed **sine tables** for trig-free motion. The highest-leverage choice is **data-driven design** — engine separate from content (level formats, tuning tables, text banks the engine interprets), making code small and content compact and changeable, and enabling the Chapter 38 toolchain. **Saving state** persists in the tier the hardware allows — disk files, cassette blocks, or (with no storage) computed-and-verified **passwords** carrying progress in the player's notebook. The deliverable is **`SKELETON99`**, the verified template — state machine, entity table, fixed-point, `lib99` resident — that every Part IX capstone instantiates, so the architecture is derived once, here, and proven on the bench. The constraint is the curriculum, and its lesson — budget, layer, densify, separate content from code — is the lesson of all good software; the TI just makes you learn it in your bones.
