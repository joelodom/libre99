## Most of a Game Is Data

Open up a finished TI game and weigh its parts, and a surprising truth emerges: the *code* is often the small part. The engine — the state machine, the entity loop, the fixed-point math (Chapter 36) — might be a few kilobytes; the *data* is most of the rest: the graphics (tile sets, sprite sheets, fonts), the levels (maps, enemy placements), the music (note streams), the text (dialogue, instructions), the tables (sine, precomputed everything — Chapter 37's table-ism). A game is mostly *content*, and content is data, and on a 32K machine that data must be made to fit — which means it must be **generated** (from human-friendly source into TI-friendly bytes), **compressed** (squeezed to fit the budget), and **budgeted** (accounted against the memory worksheet, Chapter 36). That machinery — the tools that turn a PNG into tile bytes, a tracker song into a note stream, a text file into packed prose, and the decompressors that unpack them on the console — is the **asset pipeline**, and it is the subject of this final Part VIII chapter.

The pipeline is where the **two-machine workflow** this book has practiced all along becomes explicit and industrial. You do not draw tile graphics by hand-typing `DATA` statements (you draw them in an image editor and *convert*); you do not compute a sine table by hand (you *generate* it); you do not pack text bit-by-bit manually (a *tool* does). The modern computer is the content factory — running the generators, the converters, the compressors — and the TI is the consumer, running lean decoders that unpack the factory's output at load time. This is exactly how console and embedded development works today (a powerful host builds assets for a constrained target), and it is how a 32K game holds far more content than 32K of hand-authored data ever could: the content is authored richly, compressed hard, and unpacked cheaply. This chapter builds both ends — the host-side generators (in the project's honest toolchain) and the target-side decompressors (verified on the bench) — and closes the software-engineering part on the practical craft of *fitting the content in*.

## The Bridge: The Asset Pipeline Is Universal

Every game you have ever played shipped through an asset pipeline, and most software besides. A modern game's build turns `.psd` and `.fbx` and `.wav` source files into platform-specific compressed formats (block-compressed textures, packed meshes, streamed audio), bundles them, and the runtime unpacks them on demand — a content pipeline that is often larger and more complex than the game's own code. Embedded systems generate lookup tables and configuration blobs at build time; web apps compile and bundle and minify; compilers themselves are asset pipelines for code. The pattern is universal: **author in a rich, human-friendly form on a powerful machine; transform to a lean, machine-friendly form; consume efficiently on the target.** The transformation — generation, compression, packing, bundling — is a first-class part of building software, not an afterthought.

The TI's pipeline is that universal pattern at a scale you can hold whole: the "rich source" is a PNG or a text file or a tracker song; the "transform" is a converter you can read (or write); the "lean form" is a compressed byte stream; the "consumer" is a decompressor of a few hundred bytes you can single-step. Studying it teaches the *whole shape* of content pipelines — why you separate authoring from runtime format, why you compress, why the generator and the runtime must agree on the schema — on a machine where each stage is small enough to understand completely. And it teaches the crucial discipline the big pipelines also need and sometimes botch: the **single source of truth** (§38.5), so the generator's output and the consumer's expectations never drift apart. When you have built a TI asset pipeline — a converter, a compressor, a decompressor, a schema, a budget report — you understand the content pipeline of a modern game engine, minus the scale, in full view.

## 38.1 The Two-Machine Workflow, Matured

The workflow has two machines with two jobs. The **host** (your modern computer) runs the **generation tools**: converters that turn source assets into TI byte streams, table generators, compressors, and the build orchestration that runs them in order — the *conveyor belt* from human-authored source to the binary blobs and `DATA` statements the assembler consumes. The **target** (the TI) runs the lean **consumers**: the decompressors and loaders that unpack the host's output at run time. The host does the expensive, one-time work (convert, compute, compress); the target does the cheap, repeated work (unpack, use).

Here the book's honest ground shapes the tooling (R-12). The outline imagines a `make`-driven pipeline of Python tools, and on a machine with Python that is the natural choice — but the **PC workstation this project builds on has no Python and no `make`** (the repo policy is `sh` + `cargo` only, so the toolchain runs on both workstations). So the project's pipeline is built from what *is* present and portable: the `libre99asm` and `libre99gpl` assemblers (Rust/`cargo`), `sh` scripts as the conveyor belt (`setup.sh`, `verify.sh`, and per-asset build scripts), and small `cargo`-built converters where a real program is needed — a `sh`+`cargo` pipeline that runs identically on the PC and the Mac. Where a task genuinely wants Python (rich image conversion is the classic case — reading a PNG, quantizing to the palette, tiling), that stage runs on the **Mac** (which has Python), exactly as Chapter 6's object-format work does, and the book notes the split rather than pretending one machine does all. The *principle* is machine-independent (host generates, target consumes); the *implementation* honors the repo's `sh`+`cargo` constraint, with Python-where-present for the image-heavy stages. The conveyor belt is real either way — a build command that turns source into TI bytes — and the discipline is the same: automate the transformation so a content change is a rebuild, not a hand-edit.

## 38.2 Graphics Conversion

The most common generation task is turning a **PC image** into TI graphics, and it is a matter of mapping a modern image onto the 9918's constraints (Part III). For **Graphics I tiles**, a converter reads an image, snaps it to the mode's rules — 8×8 character patterns, the two-colors-per-8×1-row color constraint (Chapter 15), the 256-tile pattern table — and emits the pattern and color bytes, deduplicating identical tiles (a tile map plus a tile set, far smaller than a full bitmap). For **bitmap thirds** (Graphics II, Chapter 15), it splits the image into the mode's three-way structure and emits the pattern/color tables per third. For **sprite sheets**, it slices an image into 8×8 or 16×16 sprite patterns and emits the pattern table (Chapter 16). The converter's job is the tedious, error-prone mapping from "a picture" to "the exact bytes the VDP tables need under the mode's constraints" — precisely the work you do *not* want to do by hand, and precisely what a tool does reliably. Map editors (Magellan-class tools for the TI) export level layouts that a converter turns into the level format (§38.5) `SKELETON99`'s loaders read (Chapter 36). This is the image-heavy stage that wants Python on the Mac (PNG decoding, palette quantization); its *output* — pattern bytes, color bytes, tile maps — is plain data the `sh`+`cargo` pipeline assembles and the bench verifies (Part III's graphics oracles confirm the converted tiles render correctly). Author the art in an image editor; convert to VDP bytes with a tool; verify the render on the bench: the graphics pipeline in three honest stages.

## 38.3 Compression That Earns Its Keep at 3 MHz

Compression on a 3 MHz machine is a *trade*: it saves ROM/RAM but costs decode time, so it "earns its keep" only when the space saved matters and the decode is fast enough — which rules out heavy schemes and rewards simple, fast ones. Three earn their keep.

**RLE (run-length encoding)** is the workhorse, and this chapter's *verified* decoder. It squeezes runs of repeated bytes (common in tile graphics, cleared regions, and sparse maps) into a count and a byte, with literals passed through. `rlelib`'s PackBits-style stream is a series of control-byte-led blocks — `C = 0` ends, `C ≥ >80` is a run of `C & >7F` copies of the next byte, `C < >80` is a literal run of `C` bytes — and its decoder is a dozen instructions, decompressing straight into RAM or VRAM. Verified on the bench: the stream `>85 >41 / >03 >42 >43 >44 / >00` (run 5×'A', literal "BCD", end) decompresses to exactly `AAAAABCD`. RLE is *where it is enough* — run-heavy data (screens, tile maps, simple graphics) — and its decoder is so small and fast (a few cycles per output byte) that it always earns its keep there. **An LZ77-family decoder** handles data with *repeated substrings* rather than just runs (text, structured data): the stream mixes literals with back-references (`copy L bytes from D back in the already-decompressed output`), so a repeated phrase or pattern is a short reference. It compresses better than RLE on such data but its decoder is larger and slower (following back-references, size and speed **measured** against the ratio it buys), so it earns its keep for larger, substring-rich assets where the extra compression pays for the extra decode — a real trade you measure per asset. **Tiny-alphabet text packing** exploits that game text uses a small character set: packing to **5 bits per character** (32 symbols — enough for A–Z plus a few) instead of 8 saves 37.5% on all prose, and adding **digraphs** (common letter pairs as single codes) squeezes RPG-scale text further — decoded by a small unpacker that shifts bits out and expands digraphs. For a text-heavy game (Chapter 41's RPG), packed prose is the difference between the script fitting and not. The rule across all three: **match the compressor to the data** (RLE for runs, LZ for substrings, bit-packing for text), and **keep the decoder small and fast** (it runs on the target, in the frame or the load, so its cost is real) — compression that earns its keep is compression whose space saved exceeds its decode cost, measured, per asset.

## 38.4 Music and Speech Assets

Sound and speech are data too, with their own generation paths, integrated into the same build. **Music** follows the sound pipeline of Chapter 19: a song authored in a tracker (or captured as a **VGM** register-write log) is converted to the TI's compact note-stream format — a sequence of "set channel N to frequency F, volume V, for duration D" events the sound driver plays back — and that conversion (VGM or tracker export → TI note stream) is a host-side generator feeding the build. **Speech** follows Chapter 20: LPC parameter frames (the vocal-tract data the synthesizer speaks) are built from source audio into the compact LPC stream, on the machine that has the LPC tooling. Both are *assets* the pipeline generates and the game's drivers consume, and both integrate into the same conveyor belt — a `sh` build step that runs the converter and emits the data blob the assembler includes — so a music change is a rebuild, like a graphics change. The honest note (R-12) from Parts IV: the project *plays* the sound-chip note streams (verified, Chapter 19), but does *not* emulate the speech synthesizer (Chapter 20), so the speech-asset build produces data the real hardware or a speech-capable emulator speaks, while the sound-asset build produces data the project's PSG plays and the bench verifies. Music and speech join graphics and levels as generated, budgeted content — the whole game, authored richly off-machine and consumed leanly on it.

## 38.5 Data Schemas and the Single Source of Truth

A generated asset is a **binary blob** with a **layout** — a level format's fields, a note stream's event structure, a table's stride — and the deep discipline here is the **single source of truth**: the generator that *emits* the binary and the 9900 code that *reads* it must agree on the layout *exactly*, or the game reads garbage. The failure mode is drift: someone changes the level format (adds a field, reorders bytes) in the generator but not in the assembly that parses it, and the game breaks in a way that is maddening to debug (the bytes are "right," but read wrong). The fix is to **generate both from one definition**: the schema is defined once, and a tool emits *both* the binary asset *and* the matching **`EQU` header** (the assembly constants naming each field's offset and each value's meaning) — so the assembly code reads the binary through the *same* symbolic offsets the generator wrote, and a schema change regenerates both in lock-step. `LEVEL_EXITS EQU 4` in a generated header, used by the assembly that parses the level, guarantees the parser's offset matches the generator's layout, because both came from the one schema. **Versioning** the schema (a version byte in the blob) lets a loader reject or migrate old data (Chapter 41's versioned saves). This is the same single-source-of-truth discipline that governs every serious data format (protocol buffers, schema-generated code, versioned file formats) — define the schema once, generate the readers and writers from it, and drift becomes impossible — applied at the scale of a TI level format, where the drift bug is common and the fix (generate the `EQU` header alongside the binary) is simple and complete.

## 38.6 Automated Budget Accounting

Chapter 36's memory worksheet was a design act done by hand; here it becomes **automated**, part of the build. Because the pipeline generates the assets and knows their sizes, the build can **print a memory report** — a tally of every asset against its budget (this bank's ROM used and free, VRAM allocated, scratchpad claimed) — so you see, *at build time*, whether the content fits, in **red ink before run-time surprises**. The value is catching the overflow early: without it, you discover the game does not fit when it crashes or corrupts at run time (a level overwriting the sound driver, a table spilling past its bank); with it, the build fails (or warns) the moment the assets exceed the budget, with a report saying exactly which budget blew and by how much. The report is a simple tool — sum the asset sizes, compare to the worksheet's limits, print the tally — but it turns memory budgeting from a hopeful hand-calculation into an enforced, automated gate, the way a modern build fails on a size regression. It closes the loop the pipeline opened: the worksheet (Chapter 36) sets the budgets, the pipeline generates the assets, and the budget report *enforces* that the generated assets fit the budgets — content, compression, and accounting, unified in the build. When the report is green, the game fits; when it is red, you compress harder or cut content, *now*, at build time, not at the demo that crashes.

## Lab 38 — Crunch the Asset Pack

The lab is the pipeline end to end, in `code/ch38/`.

**The verified decoder (`rlelib`)** is the chapter's anchor: the RLE decompressor, proven byte-exact on the bench.

```sh
libre99asm code/ch38/rledec.a99 --format bin -o build/RLE.bin --symbols build/rle.map.json
# bench:  load / pc START / x 200 / m 2000 8   ->  41 41 41 41 41 42 43 44  ("AAAAABCD")
```

Runs and literals both decode; the decoder is the few hundred bytes the whole pipeline's compression rests on.

**Crunch a 24K asset pack** is the productionization: take a realistic asset pack — tiles, two level maps, three songs, and 4 KB of text — totaling **24 KB of raw data**, run it through the pipeline (convert the graphics, compress the maps with RLE, pack the text to 5-bit, generate the note streams), and fit it into a **12 KB cartridge bank** plus a **resident 600-byte decompressor** — a 2:1 squeeze, with the budget report proving it fits. It is the whole chapter applied: generation (graphics/music converted), compression (RLE for the maps, packing for the text — each matched to its data, §38.3), schema (the level format with its generated `EQU` header, §38.5), and accounting (the report confirming 24 KB of content in a 12 KB bank, §38.6). The decompressor is resident (Chapter 35's resident segment — every bank needs it), and the compressed assets are banked. Building it is proving you can *fit the content*: 24 KB of game in 12 KB of ROM, because the pipeline squeezed it and the console unpacks it — the practical craft that lets a memory-starved machine hold a real game's worth of data.

> **Sidebar — Graph paper and what we owe them.** Before image editors and converters, a 1982 TI programmer designed a tile or a sprite on **graph paper**: an 8×8 grid, each square filled in or not, then read off row by row into the eight bytes of a pattern (each row's filled squares becoming the 1-bits of a byte, hand-converted to hex). A whole tile set was pages of graph paper, painstakingly transcribed into `DATA` statements by hand — and a mistake was a wrong hex digit that put a pixel in the wrong place, found only by running it. Level maps were graph paper too, hand-numbered grids transcribed into byte arrays; music was hand-timed note tables; the sine table was computed on a calculator, entry by entry, and typed in. Everything this chapter's *tools* do, they did *by hand*, with graph paper and patience and a tolerance for transcription error that is hard to imagine now. What we owe them is respect and remembrance: the converters and generators we take for granted replaced hours of meticulous, error-prone hand-work, and the games they made *despite* that friction — pixel by pixel, byte by byte, on graph paper — are monuments to a discipline the pipeline has mercifully automated away. When your converter turns a PNG into tile bytes in a second, spare a thought for the person who once did it with a pencil and a grid and got it right anyway. The pipeline is a gift; they built without it.
