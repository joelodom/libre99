## 1. Executive summary

This is a **genuinely high-quality project** — top-decile for a hobby emulator and better-documented than most commercial codebases. The architecture is sound and deliberately layered; the core is small, readable, dependency-free, and panic-free; the documentation practice (decision records with dates, root-cause write-ups of every major bug, milestone gates) is exemplary; and the original-content firmware rewrite is executed with a discipline (differential oracles, byte censuses, layout freeze gates, seeded fuzzing) that most professional firmware teams would envy.

The evaluation nonetheless found **real, fixable defects**, which is what this report is for. In order of consequence:

1. **A cluster of verified VDP fidelity bugs** (status-flag lifecycle, sprite Y wraparound, transparent-sprite coincidence) that will make a visible class of real games misbehave — collision-driven and sprite-heavy titles — despite the flagship titles working. All three are demonstrated by failing tests in `scratch/fidelity-repro/` and confirmed against Classic99 (whose status-clear and transparent-collision behaviors are marked hardware-tested; the sprite-wrap *threshold* is Classic99's best guess — see V2).
2. **A quantified timing gap on GROM access** (flat +4 cycles vs. hardware's measured 13–22) that makes GPL-interpreted code — the whole OS — run measurably fast relative to real pacing: plausibly tens of percent up to ~2× in GROM-bound stretches (unmeasured; P2.1 includes measuring it). It is invisible to the project's own differential tests *by construction*, because both sides of every differential run inside this same emulator.
3. **The Windows persistence hole**: on a native Windows launch (`HOME` unset — verified in this environment) config, logging, save states, resume, and screenshots all silently do nothing.
4. **Test-suite blind spots exactly where TMS9900/9918A emulators historically break** (X, 11 of 13 jump conditions, byte-ALU flags, shared-register autoincrement, bitmap/multicolor rendering, sprite coincidence, cycle counts) — including one actively misleading test that asserts nothing it claims (`jmp_backwards_displacement_is_signed` assembles `JMP +0`).
5. **Process gaps**: no CI, no LICENSE file despite the MIT declaration, and no freshness gate tying the committed rewritten-firmware binaries to their sources.

None of these threaten the project's foundations. The full prioritized fix plan, written to be handed to an agent, is **§12**.

## 2. What is being evaluated

A four-crate Cargo workspace plus original firmware content:

- **`crates/libre99-core`** (~4.2 k LOC src, 113 tests) — the emulator: TMS9900, TMS9918A, SN76489, TMC0430 GROM array, TMS9901/CRU/keyboard, `.ctg` cartridge loader, FD1771+DSR disk card, machine wiring, versioned save states. Pure `std`, zero deps, `#![forbid(unsafe_code)]`.
- **`crates/libre99-app`** (~4.2 k LOC, 34 tests) — winit/softbuffer/cpal desktop frontend with embedded assets (≈4.8 MB: 137 cartridges, 15 disks, ROMs, fonts), config TOML, logging, overlays, save states, media pickers.
- **`crates/libre99-asm`** (~2.1 k LOC + ~2.1 k tests) — a complete two-pass TMS9900 assembler (all 69 base opcodes), `.ctg` packager, table-driven disassembler, and the differential test harness for the console-ROM rewrite.
- **`crates/libre99-gpl`** (~3.0 k LOC, 112 tests + 36 probe examples) — GPL assembler/disassembler/decoder, byte-census tooling, and the build+verification harness for the console-GROM rewrite.
- **`original-content/`** — the from-scratch firmware: `system-roms/grom/console.gpl` (→ 24 KiB GROM image; complete, M0–M7) and `system-roms/rom/console.asm` (→ 8 KiB console ROM; in progress, M1–M3 done, M4 underway at pin time), plus the Titris cartridge (playable, gameplay-tested).

Bundled third-party material (TI ROMs, 137 commercial cartridge dumps, disk images, Classic99 as an external reference checkout) is used for emulation, comparison, and research — acknowledged as others' property in the README.

## 3. Architecture assessment

### 3.1 What is right (and worth protecting)

- **The dependency discipline is structural, not aspirational.** The "core is pure std" rule is enforced by crate boundaries; the workspace `Cargo.toml` documents *why*. The frontend takes exactly seven third-party crates, each justified in comments. This is the single most important architectural property the project has; every roadmap item should continue to respect it.
- **The `Bus` trait is the right seam.** The CPU sees only `read/write_word`, byte variants, CRU bits, and `wait_states`. That enabled 42 isolated CPU conformance tests against `FlatRam`, and later let the word-access/8-bit-multiplexer bug be fixed in exactly one place (`Tms9900Bus`) with the device-port semantics (`addr & 1` discard) documented against Classic99. The byte-as-primitive, word-as-two-transfers model in `machine.rs` is the correct hardware model.
- **Save states are done properly**: self-contained (ROM/GROM/cart/disk images embedded), magic+version header, bounds-checked reader that returns `Truncated` instead of panicking, **staged load** (decode into a scratch machine, swap on success — a corrupt file can never half-corrupt a session), and per-device sanitizers (`disk.load_state` clamps transfer cursors; `psg.load_state` masks the latched channel). This is better than many mature emulators.
- **Observability was built in, then paid off**: GROM read log + coverage bitmap, disk register/CRU trace, `peek/poke`, CPU inspector — and the docs show these tools *actually solving* the two hard boot bugs. The 36 `libre99-gpl` probe examples extend the same philosophy to firmware reconnaissance.
- **Docs are architecture**, here: `PLAN.md` (facts with sources, gates), `STATUS.md` (root-caused post-mortems), `ROADMAP.md` (feature shape rules — e.g. "core stays pure std", "UI is drawn, not toolkited"), `KNOWN-ISSUES.md` (authentic-quirk vs bug, with measurements). A future contributor can be productive in an afternoon.

### 3.2 Structural limitations to be deliberate about

- **A1. The timing model is frame-granular, and that is the ceiling on fidelity.** `run_frame()` = raise vblank flag, run ~50,000 CPU cycles, render once, synthesize audio once. There are no scanlines; VDP status bits (5S/C) are recomputed once per frame at render time; mid-frame VRAM/register changes can't be seen; PSG register writes are quantized to the audio-fill boundary. For the current library (menu-driven GPL titles) this is fine, and the choice is documented ("cycle-aware, not transistor-exact"). But three of the fidelity bugs in §4 are *entangled* with it — the status-flag lifecycle can only be made hardware-faithful if the flags are computed at a defined in-frame point rather than "whenever the frontend renders." The pragmatic evolution (no full scanline rewrite): move sprite evaluation/flag computation to `vblank()` time inside `run_frame()` (state), keep pixel production at render time (presentation). A true scanline-stepped VDP can stay a roadmap [later].
- **A2. Both sides of every firmware differential run inside this emulator.** The rewrite's oracle is TI's image *executed by `libre99-core`*. The project knows this (QUALITY-ASSESSMENT.md names it; the FONT2 chip-gap incident proved it) — but it deserves a standing mitigation, not just acknowledgment: a small periodic cross-check of the rewritten firmware on an independent emulator (Classic99/MAME/js99er, manually or scripted) and, one day, real hardware. Also note the corollary: the emulator's *own* timing errors (e.g. GROM stalls, §4 C2) are invisible to "perf parity" tests by construction.
- **A3. The toolchain crates are doing double duty.** `libre99-asm`/`libre99-gpl` are simultaneously general-purpose tools *and* the build+test harness for `original-content` (they `include_str!` the firmware sources; `libre99-gpl` depends on `libre99-asm` just to build the ROM for its tests). It works, but it couples tool releases to firmware state and bloats their test suites with firmware gates. When the ROM rewrite stabilizes, extract a `libre99-firmware` crate that owns the sources, the builds, the frozen-entry gates, and the differential suites, leaving the assemblers as clean tools.
- **A4. `Machine::mount_cartridge` layers instead of replacing.** Mounting cart B over cart A leaves A's ROM banks (if B is GROM-only) and A's GROM pages (where B has none) visible. Not user-facing today only because the frontend rebuilds the whole `Machine` per media change — an API landmine for the next embedder. Make mount reset cartridge state first, or document loudly.
- **A5. Frontend platform layer is implicit.** `$HOME`-only paths, Super-as-shortcut-modifier, fixed 16.667 ms `WaitUntil` pacing — each assumes macOS. The fixes are small (§6), but the pattern to adopt is one `platform` module owning paths/modifiers/timer policy.

## 4. Emulation fidelity findings (core, verified)

Each finding was verified against the Classic99 checkout (hardware-validated reference per project policy) and, for V1–V3/C1, **demonstrated by a failing test** in `scratch/fidelity-repro/tests/fidelity.rs`. File refs are to this repo at `cb5f005`; Classic99 refs to `C:\ClaudeShared\classic99`.

### V1 — VDP status read clears only F; hardware clears F, 5S, and C  *(high, small fix)*
`vdp.rs read_status()` does `status &= !ST_F`. The 9918A clears all three top flags on read — Classic99 `Tiemul.cpp:5179-5180`: `VDPS &= 0x1f; // top flags are cleared on read (tested on hardware)`. Related model wrinkle: `render()` clears-and-recomputes 5S/C per frame (and clears without recomputing when the display is blanked), where hardware is set-by-scan/cleared-by-read. Games that acknowledge a collision by reading status will see one collision as many.
**Fix:** clear F|5S|C (keep low 5 bits) on read; move sprite flag computation to `vblank()` (see A1) so set/clear ordering vs. the ISR's status read is faithful. Port `finding_v1_*` from the repro crate as the regression test, plus a "read clears 5S" twin.

### V2 — No 8-bit sprite-Y wraparound: sprites cannot enter from the top edge  *(high, small fix)*
`render_sprites()` computes `top = (y as i32) + 1`; Y=`>F8` yields 249, never drawn. Hardware wraps at 8 bits (Y=`>FF` ⇒ first row on line 0) — Classic99 `tivdp.cpp:2401-2403`: `yy=VDP[curSAL]+1; if (yy>225) yy-=256; // fade in from top`. Anything sliding a sprite in from the top pops instead. *Precision note:* the wraparound itself is certain; the exact `>225` cutoff is Classic99's own open question (its comment at `tivdp.cpp:2428` adds "TODO: is this right??"), so reproduce Classic99 per project policy but don't treat the boundary value as hardware-pinned.
**Fix:** adopt Classic99's exact formula (`if top > 225 { top -= 256 }`). Note the tempting alternative — `row = (line - Y - 1) & 0xFF`, compare `< screen_px` — is **not equivalent** for 32-pixel (magnified 16×16) sprites near Y≈224 and should not be used. Port `finding_v2_*`; add a Y=`>FF` case (first sprite row on line 0).

### V3 — Transparent (color 0) sprites are excluded from coincidence  *(high, small fix)*
`render_sprites()` does `if color == 0 { continue; }`, skipping collision tracking too. Hardware detects coincidence on **pattern bits regardless of color** — Classic99 `tivdp.cpp:2451`: "Even transparent sprites get drawn into the collision buffer." Invisible-sprite hitboxes (a standard TI technique) never collide.
**Fix:** keep the pattern/coincidence path for color 0 and skip only the framebuffer write. Port `finding_v3_*`.

### C1 — MPY/DIV with destination R15 wraps the second result word into R0  *(medium, tiny fix)*
`cpu.rs` uses `(d + 1) & 0xF` for the product-low/remainder word (and DIV's dividend-low read). Registers are memory: hardware writes `WP + 2*d + 2` = WP+32 for d=15, no wrap (Classic99 `cpu9900.cpp` uses address `x+2`). Demonstrated: `finding_c1_*` (low word landed in R0; WP+32 untouched).
**Fix:** compute the companion word by *address* (`reg_addr(d).wrapping_add(2)`), not register number, in MPY (write), DIV (read low, write both).

### C2 — GROM port accesses cost a flat +4 cycles; hardware stalls 13–22  *(medium; the big timing item)*
`machine.rs wait_states()` returns 4 for every multiplexed address including the GROM ports. Classic99's hardware-verified numbers (`Tiemul.cpp:5838/5874/5940/5958/6013`): read data ≈ **+19**, read address ≈ +13, address writes ≈ +15/+21, data writes ≈ +22. Since the whole OS and most cartridges are GPL fetched byte-by-byte from GROM, GPL-heavy code currently runs fast relative to hardware pacing — back-of-envelope, a GROM byte fetch costs ~22 cycles here vs ~41 on hardware (~1.8× on the fetch itself), diluted by zero-wait interpreter dispatch to **plausibly tens of percent up to ~2× in GROM-bound stretches (unmeasured — P2.1 includes measuring boot-to-title frames before/after)**. The rewrite's "perf parity" gates cannot see this (both sides run here — A2). Consequences today: game feel/menu pacing diverge from real hardware; consequences tomorrow: any CPU-timed firmware work (cassette) would be built on wrong time.
**Fix:** replace the boolean fast/slow `wait_states(addr)` with port-aware costs (GROM ports get the measured stalls; VDP/other mux stays ~4). Then re-baseline the empirical frame counts in integration tests (`boot.rs` 180 frames etc.) — expect boot-to-title to need more frames. Verify menu/boot behavior unchanged apart from pacing.

### C3 — Cycle-count fine points  *(low)*
STCR is charged with LDCR's formula `20+2C` (hardware: 42/44/58/60 by count); shifts with count-from-R0 miss the extra +8; DIV is flat 92 (hardware 92–124 data-dependent); `CYCLES_PER_FRAME = 50_000` vs. the 9918A's ~59.92 Hz field (≈0.13 % fast, already noted in KNOWN-ISSUES). Fix opportunistically; document what stays approximate.

### C4 — Operand evaluation order (pathological cases only)  *(low)*
`exec_dual_operand` resolves src EA (incl. autoincrement), then dst EA, then reads the source **value**; hardware reads the source value before computing the destination address. Notably, the code's own comment (`cpu.rs:429-431` — "Resolve the source, read it, then resolve the destination…") *describes the hardware order that the code below it does not implement* — evidence this is drift, not a decision. Differs behaviorally only when dest resolution mutates memory the source EA aliases (e.g. `MOV *R1, *R1+` with R1 pointing into the workspace at R1's own cell). Reorder to resolve-src → read-src → resolve-dst (Classic99's order) or fix the comment; add the shared-register autoincrement tests from §5 either way.

### C5 — Silent illegal opcodes; C6 — CRU masking; V4 — minor VDP items  *(low)*
Illegal opcodes are 6-cycle silent NOPs with no counter/hook — a wedged program is undiagnosable from outside; add a counter + accessor (core-friendly, no logging dep). `Tms9900Bus` doesn't mask CRU bit addresses to the hardware's 12 lines (FlatRam does — inconsistent). Text-mode fg=0 renders black instead of backdrop (bg side is handled); status low-5-bits don't model the last-scanned-sprite value. Details in `scratch/NOTES-core-findings.md`.

### D1 — Disk geometry is hardcoded  *(medium for anyone adding images)*
40×9×256 SS/DS with the v9t9 side-1 mirror is assumed for every image; a DSDD/80-track image LBA-mismaps silently. The VIB (sector 0) carries sectors/track, sides, density — parse it at mount with the current values as fallback, and reject size-inconsistent images. Also: disk writes mutate only the in-memory image (persist only via save-state) — fine, but document it; and the stale comment in `disk.rs command()` ("Read Address … unused by the TI DSR") contradicts the implemented `0xC0` handler right above it.

### Explicitly *not* findings
The PSG period-0 = 1024 reload matches Classic99 exactly (`sound.cpp:443`); the GROM 8K-slot wrap, destructive address read, and prefetch model match Classic99; the Graphics-II R3/R4 masking matches MAME's semantics; `KNOWN-ISSUES.md`'s screen-blank timeout is authentic firmware behavior, correctly preserved with an opt-in defeat.

## 5. Test suite assessment (`libre99-core`)

**Strengths.** The serial-chip semantics — the part of a TI-99 emulator that actually kills boots — are pinned superbly: GROM prefetch/destructive-address-read/phase-reset each have a dedicated test tied to the firmware scenario that would break; the VDP odd-half-drop regression is tested from three angles with its real-world story (`disk_controller_still_draws_the_master_title_screen`); save-state coverage includes all three malformed-input classes plus the must-not-mutate invariant and a lockstep replay; the integration gates run genuine firmware end-to-end (boot→title, cartridge menu, real DSR loading QUEST from disk). Everything is deterministic and hermetic (assets via `include_bytes!`; no sleeps, no wall-clock, no RNG).

**Gaps (ranked; each maps to action items in §12):**

1. **`X` (Execute) — zero tests.** GPL dispatch and console math rely on it; it recursively re-enters `execute` and any operand/extension-word bug corrupts the interpreter.
2. **11 of 13 conditional jumps untested** (only JMP/JEQ), and the one "signedness" test is a false comfort: `jmp_backwards_displacement_is_signed` (tests/cpu.rs:374) assembles `0x1000` = **JMP +0** and asserts only PC+2 — verified; its own comment drifts from "JMP −1" to "instead: JMP +0". The JLE/JHE/JH/JL boolean combos are the classic TMS9900 emulator bug site.
3. **Byte-ALU ops untested**: AB/SB/CB/SOCB/SZCB (carry/overflow/**parity** — CB takes parity from the *source*, AB/SB from the result), and no byte op is ever driven at an **odd address** (low-byte RMW path).
4. **Autoincrement with a shared register** (`MOV *R1+,*R1+`, `A R3,*R3+`) — the ordering pitfall CLAUDE.md itself warns about — unpinned.
5. **MPY/DIV d=15** unpinned (and currently wrong — C1).
6. **VDP Graphics II and Multicolor renderers have zero tests** — GII masking is the trickiest code in `vdp.rs` and a large fraction of the library is bitmap-mode. Sprite features beyond one 8×8 sprite (coincidence, 16×16, magnify, early-clock, priority, 5th-sprite number) untested.
7. **Untested opcode cluster**: COC, CZC, XOP (vector, R11=EA, X-bit), INV, INCT/DECT, STWP, STST, LIMI (mask changes only ever set via `set_st`), IDLE, RSET; BLWP/RTWP round-trip doesn't assert ST/R15.
8. **No cycle-count assertions at all** — a wrong constant silently shifts vblank cadence; and the empirical "settle" frame counts in integration tests are exactly what C2's fix will disturb, so a few representative `cpu.cycles()` assertions should land *with* the timing work.
9. **State sanitizer paths never executed** (corrupt-but-well-formed inputs); PSG mid-table attenuation, noise rates 0x20/0x40/follow-tone-2, N=0 reload untested; disk Step/Read-Address/Force-Interrupt/side-1 LBA untested at unit level; `@A(R0)` (R0-is-not-an-index) unpinned.
10. **Suite structure nit:** the 10 pure-`Vdp` tests live in the one file that embeds console ROMs for a single `Machine`-level test — moving that test would let VDP unit tests build without proprietary images.

The frontend suite (33 tests) covers the pure modules well (input mapping, config clamping, CLI, menu matching, PNG encoder vs. known CRCs); `video::blit` (pure scaling math, easily wrong) and frame-pacing arithmetic are untested; some tests assume `$HOME`/embedded assets (same portability issue as the app).

## 6. Frontend (`libre99-app`) findings

Verified highlights of the specialist review (full detail in the review notes; fixes in §12):

- **F1 (high) — All persistence silently dead on native Windows.** `config.rs:163-166` resolves only `$HOME`; on this Windows box PowerShell has `HOME` empty and `USERPROFILE` set (verified live). Result on a native launch: no config read/write, no log file, no save state, no resume, no screenshots — all silent. Fix: `var_os("HOME").or_else(|| var_os("USERPROFILE"))` (or platform module, A5).
- **F2 (medium) — Panic paths on device loss**: `surface.resize/buffer_mut/present().unwrap()` in the present path; window/context creation unwraps in `resumed()`; **`Mutex::lock().unwrap()` inside the cpal real-time callback** (poisoned lock ⇒ panic in the audio thread; contention ⇒ potential glitch). Fix: skip-frame-on-error in present; `try_lock` + silence (or an SPSC ring) in the callback.
- **F3 (medium) — Cmd/Super shortcuts unusable on Windows** (`Win+S`/`Win+D`/`Win+Q` are OS-reserved): screenshot, CPU inspector, quit have no working binding there. Fix: Ctrl on non-macOS or F-key equivalents.
- **F4 (medium) — Media selection desyncs after auto-resume**: machine state is restored from the snapshot but `cart_sel`/`disk_sel` are computed from config defaults — title bar lies and F2/F3/F4 cycle from the wrong baseline. Persist the selections with the state.
- **F5 (low) — Turbo sticks across focus loss** (`Focused(false)` releases keys but not `speed.turbo`); **exit auto-save clobbers the manual F6 slot** (single-slot design footgun); config rewrite drops user comments/unknown keys; `build.rs` panics on a missing asset dir; RS232/Speech ROMs embedded but unused (~40 KB); Windows timer granularity may quantize the 16.667 ms `WaitUntil` pacing (needs `timeBeginPeriod` if Windows becomes first-class).
- **Genuinely good**: pure-logic separation, self-syncing ~50 ms audio queue with 1 s overflow guard and silent underrun, no busy-spin pacing with backlog clamp, focus-loss key hygiene with exact release replay, resilient config parsing, hand-rolled PNG encoder validated against known vectors.

## 7. Toolchain crates (`libre99-asm`, `libre99-gpl`)

### 7.1 Quality
Both are genuinely good: a complete two-pass TMS9900 assembler (all 69 base opcodes, E/A-authentic left-to-right expressions, absolute-image mode with a per-byte `AORG` overlap guard), a disassembler driven by the *same* ISA table and kept honest by an assemble→disassemble→reassemble round-trip test; on the GPL side a curated, execution-validated encoder plus a full-256-opcode decoder for reconnaissance, and the byte-census machinery that turns "how compatible are we" into a failing test. Diagnostics carry 1-based line numbers with actionable messages. Zero third-party deps in both.

### 7.2 Issues
- **T1 (medium) — No staleness gate on the committed firmware binaries.** `console-rom.bin`/`console-grom.bin` are git-tracked and are what `--system-rom/--system-grom` loads, but every test builds the firmware *in memory* from source; nothing compares the committed bytes to a fresh build. An un-rebuilt `.bin` ships silently (PROGRESS.md's "rebuild manually" rule is the only guard). Fix: one test per artifact — `assert_eq!(include_bytes!(...bin)[..], build_console_rom()[..])`. **Timing caution: land this only when the sibling ROM session is idle**, since it will be red whenever that session's source is ahead of its committed binary.
- **T2 (low-med) — `libre99-gpl` two-operand immediates aren't range-checked** (byte-form `>1FF` silently truncates) while every other data path is. Mirror `dir_byte`'s check.
- **T3 (low-med) — Front-end duplication**: `string_operand`, `Diag`, `EQU`/`BYTE`/`DATA`/`TEXT`/`BSS` handling are near-copies across the two assemblers (`lex`/`expr` are shared; the rest should be).
- **T4 (low)** — `libre99-asm/src/isa.rs` and `lib.rs` headers still describe a "tracer-bullet subset / deferred formats" that the table below them fully implements; `COPY` include diagnostics lose per-file line numbers (latent — unused by `console.asm`); `census::parse_ranges` silently skips malformed `>hhhh..>hhhh` entries; a vestigial always-false `Decoded::approx` field.

### 7.3 The observed `gpl_fuzz` failure is not a defect
During the evaluation run, `gpl_fuzz::fuzz_fast` failed (seed 1, `>8340 auth=FC ours=05`). ## 8. Original content (the from-scratch firmware + Titris)

Assessed as **code**, read-only at `cb5f005` (three specialist reviews: ROM-as-code, GROM-as-code, methodology/docs).

### 8.1 What's excellent
- **Methodology.** Recon→spec→rewrite→differential-verify, with confidence-tagged fact dossiers (RECON), byte censuses bounding the compatibility surface (surface maps with a census *gate* that fails the build on unclassified authentic-only runs), ~130 differential microtests + FMT cases diffing scratchpad/VDP/VRAM against the authentic image, pixel-identity boot gates, a `[TI_ROM, OUR_ROM]` firmware matrix, seeded differential fuzzing that has already caught real bugs (SHCNT mask, MOVE→SPEC dispatch), a 137-cart differential "health panel," and 23 frozen public entry addresses enforced by a layout gate plus a per-byte overlap guard. ## 9. Stability & robustness summary

- **Core:** effectively panic-free in production paths (the one non-test `unwrap` at `disk.rs:183` is guarded by the `dsr.is_none()` early return at the top of `read_byte`, `disk.rs:171-173` — still worth rewriting as `if let` for lint-ability). - **Frontend:** the panic paths and the Windows persistence hole (§6 F1/F2) are the stability story; everything else is polish.
- **Data safety:** disk writes are volatile (in-memory only, persisted only through save-states) — acceptable if documented; the single save slot being overwritten by exit auto-save (F5) is the closest thing to a data-loss footgun.

## 10. Process & infrastructure

- **No CI.** With 400+ tests across four crates, clippy-clean discipline, and *two parallel development sessions*, CI is the highest-leverage missing piece — it would also host the T1 staleness gates and (later) an independent-emulator smoke check. Even a single GitHub Actions workflow (`cargo test --workspace && cargo clippy --workspace -- -D warnings`, Windows + macOS matrix) changes the risk profile. Caveat: the repo embeds copyrighted ROMs/cartridges into test binaries — fine for a private repo; a public CI would want that inventory private.
- **No LICENSE file** at root, while `Cargo.toml` declares MIT and README §"License and provenance" describes MIT + third-party-property carve-outs. Add the MIT text (and keep the carve-out language); if the repo is ever public, revisit distributing the ROM/cartridge inventory at all.
- **`cargo-mutants` artifacts are gitignored** — someone ran mutation testing; institutionalize it occasionally against `libre99-core` (the §5 gaps are exactly what mutants find).
- **Binary size:** ≈4.8 MB of embedded assets is a deliberate, documented trade — fine.
- **Two-session workflow:** worked well during this evaluation (the one collision was correctly diagnosable), but it relies on convention. The CI + staleness gates + "don't commit over a red fuzzer" norms make it robust.
